# RewriteBySoobak33

유니티 최적화 시리즈를 포함한 **어떤 시리즈의 어떤 글에든** 동일한 품질의 리라이팅을 수행하는 독립 스킬.

핵심 목표: **직관적인 이해가 가능한 자연스러운 한국어 문체의 문장들로 구성된 간결하지만 완전한 설명**을 독자에게 전달하는 것.

**독립 스킬** — 파이프라인에 속하지 않으며, 다른 스킬을 참조하거나 의존하지 않는다. 단독으로 실행한다.

## 인자

- `$ARGUMENTS`: 수정할 파일 경로. 없으면 사용자에게 파일 경로를 요청한다.

---

## 실행 원칙

**이 스킬이 호출되면 항상 Phase 1~4 전체를 빠짐없이 실행한다.** 글이 이미 깔끔해 보이거나 변경할 것이 적어 보여도, 프로세스를 축약하거나 실행이 불필요하다고 판단하지 않는다.

---

## 절대 제약 조건

모든 Phase의 모든 수정에 적용되는 불변 규칙:

1. **질문형 문체 금지** — 선언문만 사용한다.
2. **기술 용어 절대 보존** — L2/L3, 필레이트, 텍셀, 드로우콜, SIMT, IMR, TBDR 등 구체적 기술 용어를 간소화하거나 상위 개념으로 대체하지 않는다.
3. **가독성을 위한 설명 생략 금지** — 정밀도를 유지하고, 어려운 내용은 설명을 추가하여 해결한다.
4. **구체적 정보 희생 금지** — 수치, 기술 메커니즘, 단계별 상세는 반드시 보존한다.
5. **"이 글에서 다루기엔 복잡하다" 금지** — 맥락에서 등장한 개념은 완전히 설명한다. "별도 글에서 다루는 것이 적합합니다"는 절대 사용하지 않는다.
6. **초보자 대상** — 모든 암묵적 전제를 명시한다. "초보자"라고 표현할 때 특정 분야를 한정하지 않는다.
7. **교과서 표현 제거** — "~라는 것입니다", "중요한 점은", "유의해야 합니다" 등을 제거한다.
8. **개념적 글에 JSON/YAML 금지** — 개념 설명 글에 설정 코드를 삽입하지 않는다.
9. **산문 우선** — 설명에는 불릿 리스트보다 산문을 사용한다.
10. **다이어그램 결론은 중복이 아님** — 개념 다이어그램 내부의 "→" 결론 라인은 다이어그램의 자기 완결성을 위해 보존한다. 코드 출력 블록에서만 산문과 중복되는 "→" 라인을 제거한다.
11. **메타 문장 허용** — "이 글에서는 ... 다룹니다", "살펴봅니다" 등 메타 문장은 허용한다. 수정하거나 플래깅하지 않는다.
12. **마무리 섹션 형식** — `## 마무리` 제목을 사용하고, 하단에 **관련 글** → **시리즈** → **전체 시리즈** 순서로 배치한다. 마무리 섹션은 아래 3단 구조를 따른다:
    - **도입 명제 (1~2줄)**: 글의 핵심 주장을 선언문으로 제시
    - **불릿 요약 (5~8항목)**: 해당 글에서 다룬 내용 중 중요한 부분만 **경어체(~합니다/~입니다)**로 요약. 각 항목은 독립적으로 완전한 의미를 가지며, 기술 용어를 정확히 보존한다
    - **통찰 마무리 (1~2줄)**: 불릿들의 공통점을 메타 수준에서 정리하거나, 다음 글로의 자연스러운 확장을 제시
    - 도입 명제와 통찰 마무리는 **글의 실제 내용과 맥락에 부합**해야 한다. 글에서 다루지 않은 주장을 하거나, 글의 논조와 어긋나는 마무리를 작성하지 않는다

## 20가지 변경 패턴 정의

### 탐지(Detection) 패턴 — 문장 구조를 유지한 채 특정 요소만 수정

---

#### V1. ASCII 다이어그램 → 반응형 SVG 변환

**정의:** 코드 블록 안의 박스 드로잉 문자(─│┌┐└┘├┤ 등), 파이프(|), 하이픈(-), 플러스(+), 화살표(→, ->, ==>) 등으로 구성된 구조를 반응형 SVG로 변환한다.

**감지 기준:**
- 코드 블록(```) 안에 박스 드로잉 문자, 파이프+하이픈 조합, 화살표 등이 포함된 시각적 구조가 있는가?
- 단순 코드 출력/터미널 시뮬레이션이 아니라, 개념·구조·관계를 표현하는 다이어그램인가?

**수정 방향:** 코드 블록을 SVG로 교체. 단순 코드 출력은 그대로 유지.

**SVG 규칙:**
```
래퍼: <div style="text-align: center; margin: 1.5em 0;">
태그: <svg viewBox="0 0 {W} {H}" xmlns="http://www.w3.org/2000/svg" style="max-width: 620px; width: 100%;">
색상: 모든 stroke, fill에 currentColor만 사용. 하드코딩 색상(#000, black 등) 금지
배경: fill="currentColor" fill-opacity="0.06"
테두리: stroke="currentColor" stroke-width="1.5"
모서리: rx="5"
글꼴: font-family="sans-serif"
제목 텍스트: font-size="13" font-weight="bold"
보조 텍스트: font-size="9" ~ "11", opacity="0.5" ~ "0.6"
화살표: <line> + <polygon points="..."> 조합
닫기: </svg></div>
```

**전후 예시:**
- Before: ` ```\n ┌─────────┐    ┌─────────┐\n │ Vertex  │───▶│Fragment │\n └─────────┘    └─────────┘\n ``` `
- After: `<div style="text-align: center; margin: 1.5em 0;"><svg viewBox="0 0 300 60" ...> ... </svg></div>`

---

#### V2. 코드 블록 표 → HTML `<table>` 변환

**정의:** 코드 블록 안에 정렬된 열/행 구조로 작성된 데이터 표를 HTML `<table>` 또는 마크다운 테이블로 변환한다.

**감지 기준:**
- 코드 블록 안에 파이프(|)나 공백 정렬로 구성된 행/열 데이터가 있는가?
- 실제 코드 출력이 아니라 비교·정리 목적의 데이터인가?

**수정 방향:** 마크다운 테이블로 변환. 셀 내용이 복잡하면 HTML `<table>` 사용.

**전후 예시:**
- Before: ` ```\n 항목     | Built-in | URP\n 배칭     | 수동     | SRP Batcher\n ``` `
- After: `| 항목 | Built-in | URP |\n|------|----------|-----|\n| 배칭 | 수동 | SRP Batcher |`

---

#### V3. 코드 블록 수식 → LaTeX 변환

**정의:** 본문이나 코드 블록 안에 텍스트로 작성된 수학 공식을 KaTeX(LaTeX) 표기로 변환한다.

**감지 기준:**
- 텍스트로 작성된 수식(a^2, x/y, 시그마 등)이 있는가?
- 그리스 문자, 첨자, 분수, 벡터 표기가 텍스트로 되어 있는가?

**수정 방향:** 인라인 수식은 `$...$`, 블록 수식은 `$$...$$` 사용. KaTeX v0.12.0 지원 범위 내 작성.

**전후 예시:**
- Before: `드로우콜 = 오브젝트 x (1 + 추가 조명)`
- After: `$\text{드로우콜} = \text{오브젝트} \times (1 + \text{추가 조명})$`

---

#### V4. 가독성을 위한 문단 줄바꿈 조정

**정의:** 하나의 논점을 구성하는 관련 문장들이 불필요하게 빈 줄로 분리되어 있거나, 반대로 서로 다른 논점이 한 문단에 뭉쳐 있는 경우를 조정한다.

**감지 기준:**
- 빈 줄로 분리된 2~3개의 짧은 문장이 실제로는 하나의 논점을 구성하는가?
- 하나의 문단 안에 서로 다른 논점 2개 이상이 섞여 있는가?

**수정 방향:** 같은 논점은 하나의 문단으로 합치고, 다른 논점은 문단을 분리한다.

**전후 예시:**
- Before: `"URP는 모바일에 최적화되어 있습니다."\n\n"SRP Batcher를 기본으로 사용합니다."\n\n"싱글패스 방식으로 조명을 처리합니다."`
- After: `"URP는 모바일에 최적화되어 있습니다. SRP Batcher를 기본으로 사용하고, 싱글패스 방식으로 조명을 처리합니다."`

---

#### S1. 복합문 → 짧은 선언문 분리

**정의:** 3개 이상의 절이 포함된 긴 문장을 짧은 선언문으로 분리한다.

**감지 기준:**
- 하나의 문장에 3개 이상의 절(주어+서술어 단위)이 이어져 있는가?
- 문장이 50자를 넘으며 한 번에 의미가 파악되지 않는가?

**수정 방향:** 절 단위로 분리하되, 인과 관계가 있는 절은 연결을 유지한다.

**전후 예시:**
- Before: `"GPU는 프래그먼트 셰이더를 실행하여 각 픽셀의 색상을 계산하고, 이 결과를 프레임 버퍼에 기록한 뒤, 최종적으로 화면에 출력합니다."`
- After: `"GPU는 프래그먼트 셰이더를 실행하여 각 픽셀의 색상을 계산합니다. 계산된 결과는 프레임 버퍼에 기록되고, 최종적으로 화면에 출력됩니다."`

---

#### S2. 교과서 표현 제거

**정의:** 교과서나 교재에서 자주 쓰이는 딱딱하고 교조적인 표현을 자연스러운 서술로 대체한다.

**감지 기준:**
- "~라는 것입니다", "중요한 점은", "유의해야 합니다", "주목해야 할 것은", "핵심은 ~에 있습니다" 등의 표현이 있는가?

**수정 방향:** 해당 표현을 제거하고, 직접적이고 자연스러운 서술로 대체한다.

**전후 예시:**
- Before: `"중요한 점은 드로우콜이 CPU에서 발생한다는 것입니다."`
- After: `"드로우콜은 CPU에서 발생합니다."`

---

#### S3. 수동태/추상적 → 능동태/구체적 동사

**정의:** 수동태나 추상적 동사를 능동태와 구체적 동사로 교체한다.

**감지 기준:**
- "~에 의해 처리됩니다", "~이 수행됩니다", "~가 이루어집니다" 등 수동태가 사용되는가?
- "처리하다", "수행하다", "진행하다" 등 구체적 동작을 알 수 없는 추상 동사가 사용되는가?

**수정 방향:** 주어를 명시하고 능동태로 전환. 추상 동사를 구체적 동작 동사로 교체.

**전후 예시:**
- Before: `"조명 계산이 프래그먼트 셰이더에 의해 수행됩니다."`
- After: `"프래그먼트 셰이더가 각 픽셀의 조명을 계산합니다."`

---

#### S4. 주관적 표현 제거

**정의:** "놀랍게도", "매우", "상당히", "대단히" 등 주관적 강조 표현을 제거한다.

**감지 기준:**
- 객관적 사실 서술에 감탄이나 정도를 나타내는 부사/형용사가 붙어 있는가?

**수정 방향:** 주관적 표현을 삭제하고, 필요하면 구체적 수치로 대체한다.

**전후 예시:**
- Before: `"놀랍게도 드로우콜이 매우 크게 감소합니다."`
- After: `"드로우콜이 절반 이하로 감소합니다."` (수치가 있는 경우) 또는 `"드로우콜이 크게 감소합니다."` (수치 없는 경우, "매우" 삭제)

---

#### S5. 관련 짧은 문단 병합

**정의:** 같은 논점에 속하는 짧은 문단들을 하나의 흐름 있는 산문 문단으로 병합한다.

**감지 기준:**
- 빈 줄로 분리된 2~3개의 짧은 문장(각 1~2문장)이 같은 논점을 다루는가?
- 합치면 흐름이 더 자연스러운가?

**수정 방향:** 접속사나 연결어를 사용하여 하나의 문단으로 병합한다.

**전후 예시:**
- Before: `"Built-in은 멀티패스 방식입니다."\n\n"각 조명마다 별도의 드로우콜이 발생합니다."\n\n"따라서 조명이 많을수록 비용이 증가합니다."`
- After: `"Built-in은 멀티패스 방식으로, 각 조명마다 별도의 드로우콜이 발생합니다. 따라서 조명이 많을수록 비용이 증가합니다."`

---

#### S6. 접속사/연결어 개선

**정의:** 부적절하거나 부자연스러운 접속사/연결어를 맥락에 맞게 교체하고, 불필요한 접속사를 제거한다.

**감지 기준:**
- "그리고", "또한", "그래서" 등이 기계적으로 반복되는가?
- 인과 관계가 아닌데 "따라서", "그래서"가 사용되는가?
- 접속사 없이도 자연스럽게 이어지는데 접속사가 있는가?

**수정 방향:** 맥락에 맞는 접속사로 교체하거나, 불필요하면 삭제.

**전후 예시:**
- Before: `"GPU는 정점을 변환합니다. 그리고 래스터라이저가 프래그먼트를 생성합니다. 그리고 프래그먼트 셰이더가 색상을 계산합니다."`
- After: `"GPU는 정점을 변환합니다. 래스터라이저가 프래그먼트를 생성하면, 프래그먼트 셰이더가 각 프래그먼트의 색상을 계산합니다."`

### 전면 재수정(Full Rewrite) 패턴 — 내용/구조 자체를 새로운 관점에서 재구성

---

#### C1. 교육적 흐름 문제

**정의:** 정의→동기 순서가 아닌 동기→정의 순서로 재구성. 독자가 "왜 이것이 필요한가?"를 먼저 이해한 뒤 정의를 배워야 한다.

**감지 기준:**
- 정의가 먼저 나오고, 그것이 왜 필요한지가 뒤에 나오는가?
- 독자가 "이게 왜 중요한데?"라고 느끼며 읽게 되는 구간이 있는가?

**수정 방향:** 문제/동기 → 원인 → 정의 → 메커니즘 → 예시 순서로 재구성.

**전후 예시:**
- Before: `"**프러스텀 컬링(Frustum Culling)**은 카메라 시야 밖의 오브젝트를 제외하는 기법입니다. 모든 오브젝트를 렌더링하면 GPU 비용이 과다해집니다."`
- After: `"씬의 모든 오브젝트를 매 프레임 렌더링하면 GPU 비용이 급격히 증가합니다. 화면에 보이지 않는 오브젝트까지 처리할 이유가 없으므로, 카메라 시야 밖의 오브젝트를 렌더링 대상에서 제외하는 기법이 **프러스텀 컬링(Frustum Culling)**입니다."`

---

#### C2. 암묵적 전제 누락

**정의:** 초보자가 이해하기 위해 필요한 전제 지식이 설명 없이 사용되는 경우, 해당 전제를 명시적으로 설명한다.

**감지 기준:**
- 전문 용어가 정의 없이 사용되는가?
- "~이므로"의 근거가 독자에게 자명하지 않은가?
- 특정 기술이 왜 그렇게 동작하는지의 배경이 빠져 있는가?

**수정 방향:** 전제를 1~2문장으로 명시한 뒤 본론을 이어간다. 다른 글에서 이미 상세히 다룬 전제는 한 줄 요약 + 링크로 처리.

**전후 예시:**
- Before: `"TBDR은 타일 단위로 프래그먼트를 처리합니다."`
- After: `"모바일 GPU는 대역폭을 절약하기 위해 화면을 작은 타일로 나누어 처리합니다. **TBDR(Tile-Based Deferred Rendering)**은 이 타일 단위 구조 위에서, 각 타일 내의 프래그먼트를 한꺼번에 처리하는 방식입니다."`

---

#### C3. 인과 관계 미명시

**정의:** "A하면 B하다"에서 A→B의 인과가 명시되지 않아 독자가 왜 그런지 알 수 없는 경우, 인과 고리를 명시한다.

**감지 기준:**
- "~하면 ~가 발생합니다"에서 중간 메커니즘이 생략되어 있는가?
- 효과/결과만 제시되고 원인이 없는가?

**수정 방향:** 중간 메커니즘을 1~2문장으로 보충하여 인과 체인을 완성한다.

**전후 예시:**
- Before: `"오버드로우가 많으면 성능이 저하됩니다."`
- After: `"오버드로우가 발생하면 같은 픽셀에 대해 프래그먼트 셰이더가 여러 번 실행됩니다. 셰이더 실행 횟수가 늘어나면 GPU의 연산량과 메모리 대역폭 소비가 모두 증가하므로, 성능이 저하됩니다."`

---

#### C4. 추상적 서술 → 구체적 예시/수치 필요

**정의:** 추상적이고 모호한 서술을 구체적인 예시, 수치, 시나리오로 보강한다.

**감지 기준:**
- "많은", "큰", "상당한" 등 정량화 가능한 내용이 정성적으로만 서술되어 있는가?
- 원리 설명은 있지만 구체적 예시가 없어 독자가 와닿지 않는가?

**수정 방향:** 가능하면 구체적 수치를, 수치가 없으면 구체적 시나리오/예시를 추가한다.

**전후 예시:**
- Before: `"배칭을 사용하면 드로우콜이 크게 줄어듭니다."`
- After: `"예를 들어, 동일한 머티리얼을 사용하는 오브젝트 100개가 있다면, 배칭 없이는 100회의 드로우콜이 발생합니다. 배칭을 적용하면 이를 1~2회로 줄일 수 있습니다."`

---

#### C5. 섹션 재구조화

**정의:** 섹션의 순서가 학습 진행에 맞지 않거나, 한 섹션 안에 이질적 주제가 섞여 있는 경우 구조를 재편한다.

**감지 기준:**
- 하위 개념이 상위 개념보다 먼저 설명되는가?
- 한 섹션 안에서 갑작스러운 주제 전환이 발생하는가?
- 섹션 순서를 바꾸면 독자의 이해도가 향상되는가?

**수정 방향:** 학습 진행 순서(기초→응용, 원인→결과, 문제→해결)에 맞게 섹션을 재배치하거나 분리한다.

**전후 예시:**
- Before: `"## 멀티패스의 비용 계산" 안에 비용 공식 → 비용 표 → 갑자기 "조명 품질 등급" 설명`
- After: `"## 멀티패스의 비용 계산" (비용 공식 + 표) → "## 조명의 품질 등급" (별도 섹션으로 분리)`

---

#### C6. 기술 상세 확장

**정의:** 암묵적으로 전제된 기술 메커니즘을 단계별로 풀어서 설명한다. 원문에 "A가 B를 한다"만 있고 중간 단계가 생략된 경우 보충한다.

**감지 기준:**
- 기술적 동작이 하나의 문장으로 축약되어, 실제로는 여러 단계를 포함하는데 생략된 것은 아닌가?
- 독자가 "구체적으로 어떻게?"라고 물을 만한 서술이 있는가?

**수정 방향:** 생략된 중간 단계를 보충하되, 글의 범위와 깊이에 맞는 수준으로 작성한다.

**전후 예시:**
- Before: `"SRP Batcher는 셰이더 바인딩 비용을 줄입니다."`
- After: `"GPU가 오브젝트를 그리려면 해당 오브젝트의 셰이더를 바인딩해야 합니다. 셰이더가 바뀔 때마다 GPU 파이프라인 상태를 재설정하는 비용이 발생합니다. **SRP Batcher**는 같은 셰이더 변형(Shader Variant)을 사용하는 오브젝트들을 연속으로 그려서, 셰이더 바인딩 횟수를 줄입니다."`

---

#### T1. 용어 통일

**정의:** 같은 개념을 가리키는 여러 표기를 하나로 통일한다.

**감지 기준:**
- 같은 개념이 "드로우콜", "Draw Call", "드로우 콜" 등 여러 표기로 쓰이는가?
- 한글 음차와 영어 표기가 혼용되는가?

**수정 방향:** 최초 등장 시 `**한글 용어(English Term)**` 형태로 소개하고, 이후에는 한글 용어로 통일한다. 볼드+영문 소개는 최초 1회만.

**전후 예시:**
- Before: `"드로우콜이 증가합니다." ... "Draw Call 수를 줄여야 합니다." ... "드로우 콜 배칭을 사용합니다."`
- After: `"**드로우콜(Draw Call)**이 증가합니다." ... "드로우콜 수를 줄여야 합니다." ... "드로우콜 배칭을 사용합니다."`

---

#### T2. 기술 용어 보존 + 역할 정의

**정의:** 기술 용어를 절대 간소화하지 않고, 초보자를 위해 역할/맥락을 추가한다.

**감지 기준:**
- 기술 용어가 등장했지만 역할이나 맥락 없이 이름만 던져진 곳이 있는가?
- 기술 용어가 상위 개념으로 대체되어 정확성이 떨어진 곳이 있는가?

**수정 방향:** 용어를 유지하면서 역할을 한 절로 추가한다. 절대 간소화하지 않는다.

**전후 예시:**
- Before: `"SIMT 구조에서 실행됩니다."`
- After: `"GPU의 **SIMT(Single Instruction, Multiple Threads)** 구조에서 실행됩니다. SIMT는 하나의 명령어를 수십 개의 스레드가 동시에 실행하는 방식으로, GPU가 대량의 병렬 연산을 처리하는 핵심 구조입니다."`

---

#### T3. 다이어그램-산문 통합

**정의:** 다이어그램과 산문이 서로를 보완하도록 통합한다. 다이어그램은 자기 완결적으로, 산문은 다이어그램에 없는 맥락·해석을 제공하도록 조정한다.

**감지 기준:**
- 다이어그램 직후의 산문이 다이어그램의 내용을 그대로 반복하는가?
- 다이어그램이 산문 없이는 이해 불가능한 불완전한 구조인가?

**수정 방향:** 다이어그램에 필요한 레이블/결론 라인을 추가하여 자기 완결적으로 만들고, 산문에서는 다이어그램이 보여주지 않는 "왜", "의미", "영향" 등을 설명한다.

**전후 예시:**
- Before: (다이어그램에 "A → B" + 산문에 "A에서 B로 전달됩니다" — 동일 내용 반복)
- After: (다이어그램에 "A → B → 결과: C" 자기 완결 + 산문에 "이 과정에서 B의 부하가 병목이 됩니다" — 새로운 맥락 추가)

---

#### T4. 주어 명시 및 인과 언어 강화

**정의:** 생략된 주어를 명시하고, 인과 관계를 나타내는 언어를 강화한다.

**감지 기준:**
- 문장의 주어가 생략되어, 무엇에 대한 이야기인지 파악하려면 앞 문장을 다시 읽어야 하는가?
- "~하면 ~됩니다"에서 주체가 모호한가?
- 인과 관계가 단순 나열로 표현되어 있는가?

**수정 방향:** 주어를 명시하고, 나열("그리고", "또한")을 인과("~하면", "~하므로", "이 결과")로 교체한다.

**전후 예시:**
- Before: `"같은 포워드 렌더링이지만, 조명을 몇 번의 패스로 처리하는지가 다릅니다."`
- After: `"**Built-in과 URP**는 모두 포워드 렌더링을 사용하지만, 조명을 몇 번의 드로우콜로 처리하는지가 다릅니다."`

## Phase 1: 다중 스케일 분석 + 시리즈 개념 맵 구축 (메인 에이전트)

### 1-A. 시리즈 개념 맵 구축

대상 파일 수정 전, 시리즈 전체 글을 읽고 **개념 인벤토리**를 구축한다.

#### 시리즈 목록 자동 추출 방법

1. `$ARGUMENTS` 파일을 `Read`로 읽는다.
2. 하단의 `**전체 시리즈**` 마크다운 섹션을 찾는다.
3. 해당 섹션의 링크 목록(예: `[제목](/dev/unity/XXX/)`)에서 URL 경로를 추출한다.
4. URL 경로를 실제 파일 경로로 변환한다:
   - URL: `/dev/unity/2026/02/15/MemoryManagement-1/`
   - 파일: `dev/unity/_posts/2026-02-15-[MemoryManagement-1].md`
   - 변환 규칙: URL의 날짜 경로를 `-`로 연결하고, 시리즈명을 `[]`로 감싸서 `_posts` 디렉토리에서 찾는다.
   - URL에서 직접 변환이 어려우면, `Glob`으로 시리즈명을 기반으로 탐색한다.
5. `(현재 글)` 표시가 있는 항목으로 대상 파일의 시리즈 내 위치를 파악한다.
6. 시리즈의 모든 글을 `Read`로 읽는다.

이 방식은 시리즈를 하드코딩하지 않으므로, 어떤 시리즈든 동일하게 동작한다.

#### 개념 맵 구축

시리즈 전체를 읽고 아래 구조의 개념 맵을 만든다:

| 항목 | 설명 |
|------|------|
| **개념명** | 기술 개념 또는 용어 (예: 드로우콜, 프러스텀 컬링, TBR) |
| **최초 설명 위치** | 어느 글의 어느 섹션에서 처음 상세 설명되는가 |
| **이후 등장 위치** | 이후 글에서 다시 언급되는 모든 위치 |
| **등장 유형** | `정의(Definition)` / `맥락 요약(Context Recap)` / `전제 참조(Prerequisite Reference)` / `불필요 반복(Redundant)` |

### 1-B. 5개 스케일 통독 분석

대상 파일을 5개 스케일로 통독하며 모든 이슈를 분류한다.

| 스케일 | 점검 항목 |
|--------|----------|
| **글 전체** | 주제 완결성, 시리즈 내 위치와 역할, 도입-전개-마무리 흐름 |
| **섹션** | 섹션 순서의 교육적 흐름, 섹션 간 연결, 중복 배치 |
| **문단** | 단일 책임(하나의 문단 = 하나의 개념), 문단 간 인과 관계 |
| **문장** | 선언적 문체, 주어 명시, 능동태, 교과서 표현 제거, 길이 |
| **단어** | 용어 통일, 기술 용어 보존, 주관적 표현 제거, 조사/접속사 |

#### 분류 기준

각 이슈를 **탐지(Detection)** 또는 **전면 재수정(Full Rewrite)**으로 분류한다:

- **탐지(Detection)** — V1~V4, S1~S6: 문장 구조를 유지한 채 특정 요소만 수정
- **전면 재수정(Full Rewrite)** — C1~C6, T1~T4: 내용/구조 자체를 새로운 관점에서 재구성

한 문장에 두 유형 이슈가 모두 있으면: 서브에이전트 1이 탐지 수정 먼저 적용 → 서브에이전트 2가 해당 문장을 재수정 시 반영.

## Phase 2: 이중 서브에이전트 실행 (순차)

두 서브에이전트 모두 **파일 전체를 처음부터 끝까지 꼼꼼히 통독**한다. 순차 실행으로 충돌을 방지하면서도 각 에이전트가 전체 파일을 빠짐없이 점검한다.

### 서브에이전트 1 — 탐지 수정 (Detection) [먼저 실행]

`Agent`(subagent_type=general-purpose, model=opus)로 호출한다.

**서브에이전트 1 프롬프트 템플릿:**

```

파일 전체를 처음부터 끝까지 한 줄도 빠짐없이 읽으며, 아래 10가지 탐지 패턴을 적용합니다.

{파일 경로}

## 절대 제약 조건

2. 기술 용어 절대 보존 — 구체적 기술 용어를 간소화하거나 상위 개념으로 대체 금지

### C1. 교육적 흐름 문제

5. "이 글에서 다루기엔 복잡하다" 금지
6. 초보자 대상 — 모든 암묵적 전제 명시
7. 교과서 표현 제거
8. 개념적 글에 JSON/YAML 금지
9. 산문 우선
10. 다이어그램 결론은 중복이 아님
11. 메타 문장 허용
12. 마무리 섹션: ## 마무리 + 관련 글 → 시리즈 → 전체 시리즈. 마무리 본문은 3단 구조: (1) 도입 명제 1~2줄 (2) 불릿 요약 5~8항목 — 글의 중요한 내용만 경어체로 요약, 기술 용어 보존 (3) 통찰 마무리 1~2줄. 도입 명제와 통찰 마무리가 글의 실제 내용·맥락에 부합하는지 점검

## 7개 품질 차원

### 1. 직관적이고 자연스러운 이해
각 문장이 읽는 즉시 의미가 파악되는가? 돌려 읽어야 하는 문장은 없는가?

### 2. 자연스러운 한국어 문체
번역투가 아닌 자연스러운 한국어인가? 문맥 흐름이 끊기지 않는가?
- 수동태 잔존 점검
- 주어-서술어 거리 점검
- 병렬 구조의 자연스러움 (각 절이 서술어를 갖추었는가)
- 종결어미 단조로운 반복 점검

### 3. 간결하지만 완전한 설명
불필요한 수식어 없이도 필요한 정보가 모두 담겨 있는가?
- 동어 반복 점검 (같은 의미의 다른 표현 반복)
- 섹션 경계 동어 반복 (새 섹션 첫 문장이 이전 섹션 결론 반복)
- 소제목·도입문 중복 (소제목이 전달한 신호를 도입문이 반복)

### 4. 시리즈 중복 배치
이전/이후 글과 겹치는 설명이 적절한 위치에 있는가? 불필요한 반복은 없는가?
- 개념 맵 기반으로 점검
- 같은 깊이의 중복 설명 잔존 여부

### 5. 주제의 완전한 이해
이 글만 읽고도 주제를 완전히 이해할 수 있는가? 빠진 전제가 없는가?
- 전문 용어가 정의 없이 사용되는 곳
- 인과 관계가 빠진 곳
- 독자가 "구체적으로 어떻게?"라고 물을 곳

### 6. 사실관계 검증
기술 용어, 수치, 메커니즘 설명이 정확한가?
- 데이터 흐름, 동작 순서, 인과 관계의 기술적 정확성
- 일반 원리와 특정 엔진 구현의 구분
- 존재하지 않는 용어 생성 여부

### 7. 직관적 이해
복잡한 개념이 단계적으로 풀려서 "아, 그렇구나" 하고 와닿는가?
- 추상에서 구체로의 전개
- 비유/예시의 적절성
- 문제→원인→해결의 흐름

### 8. 마무리 섹션 품질
## 마무리 섹션이 3단 구조를 갖추고, 글의 실제 내용과 정합하는가?
- **도입 명제 점검**: 글의 핵심 주장을 선언문으로 정확히 반영하는가? 글에서 다루지 않은 내용을 주장하거나, 글의 논조와 어긋나지 않는가?
- **불릿 요약 점검**: 글에서 다룬 내용 중 중요한 부분만 선별하여 경어체(~합니다/~입니다)로 요약되어 있는가? 기술 용어가 정확히 보존되어 있는가? 글에 없는 내용이 불릿에 추가되지 않았는가? 5~8항목 범위인가?
- **통찰 마무리 점검**: 불릿들의 공통점을 메타 수준에서 정리하거나, 다음 글로의 자연스러운 확장을 제시하는가? 글의 맥락에서 벗어난 비약이 없는가?
- **전후 정합성**: 도입 명제 → 불릿 → 통찰 마무리의 흐름이 자연스럽고, 본문의 전개 순서·강조점과 일관되는가?

## 작업 방법
1. 파일 전체를 Read로 읽는다
2. 글의 첫 문장부터 마지막 문장까지 한 줄씩 읽으며, 7개 차원 중 하나라도 미달이면 해당 문장을 즉시 수정
3. 시리즈의 인접 파트를 Read로 읽어 중복 확인 (개념 맵 참조)
4. 모든 수정을 Edit로 적용
5. 수정 완료 후 파일을 다시 Read로 읽어 최종 검증:
   - 7개 차원 모두 충족되는가?
   - 기술 용어·수치가 하나도 누락되지 않았는가?
   - 수정이 새로운 문제를 만들지 않았는가?
6. 결과를 아래 형식으로 출력:

### 전문가 리뷰 결과
| 차원 | 탐지 건수 | 수정 건수 | 주요 사례 |
|------|----------|----------|----------|
| 1. 직관적·자연스러운 이해 | N건 | N건 | ... |
| 2. 자연스러운 한국어 문체 | N건 | N건 | ... |
| 3. 간결+완전한 설명 | N건 | N건 | ... |
| 4. 시리즈 중복 배치 | N건 | N건 | ... |
| 5. 주제의 완전한 이해 | N건 | N건 | ... |
| 6. 사실관계 검증 | N건 | N건 | ... |
| 7. 직관적 이해 | N건 | N건 | ... |
| 8. 마무리 섹션 품질 | N건 | N건 | ... |

### 주요 변경 (Before/After)

|------|------|--------|-------|------|
