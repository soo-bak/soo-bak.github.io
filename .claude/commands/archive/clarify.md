# 이해 — Clarify (Stage 2/4)

독자가 외부 지식 없이 글만으로 개념을 이해하도록 설명을 보강합니다. 빠진 전제, 추상→구체 연결, 단계 서술, 개념 접지.

## 파이프라인 위치

```
/foundation → ▶ /clarify → /polish → /verify
   기반          이해         다듬기      검증
```

## 인자
- `$ARGUMENTS`: 수정할 파일 경로

## 적용 axis (memory)
- 1순위: **화용론** (`linguistic_pragmatics.md`)

## 핵심 원칙

**"독자가 모른다고 가정하되, 바보라고 가정하지 않는다."**

독자는 기술 용어를 모를 수 있지만 일상적 경험은 가지고 있다. 그 경험에서 출발하여 새 개념으로 연결.

## 점검 영역

### A. 빠진 전제 보강 (화용론 3)
- 독자가 알아야 할 전제는 발화 전에 제시
- 시리즈 이전 글에 정의된 개념은 1줄 요약 후 사용
- "이해 안 됨" 사용자 피드백 → **빠진 전제 점검 우선**

### B. 추상 → 구체 연결 (개념 접지)
- 새 개념 첫 등장 시 일상 경험 / 이미 다룬 개념으로 비유·대응
- 구체 예시 먼저, 추상 원리 도출은 그 다음 (화용론 2)
- 비유로 시작하더라도 한계점 명시 ("비유는 X까지만, 정확히는 Y")
- 비유가 잘못된 멘탈모델 유발 시 즉시 정확한 정의로 보강

### C. 단계별 동사 서술 (화용론 2)
- 단계별 메커니즘을 라벨로 압축 X (예: "번들을 메모리로 불러오고, 압축 해제하고, 객체로 직렬화하고" → "번들 로드"로 압축 금지)
- 초보 독자는 단계를 따라가며 이해

### D. 결과·영향 명시 (화용론 9)
- 행동만 서술하고 결과 없으면 "그래서?" 발생
- "X합니다" → "X해서 Y가 됩니다"로 결과 연결, 또는 삭제 검토

### E. 지시어 일의성 (화용론 1)
- 모든 "이/그/저/이런/그런"이 직전 문장에서 일의적으로 해석되는지
- 후보 둘 이상이면 구체 명사로 교체
- "이/그" 체인 2회 이상 연속 X

### F. 메타 framing의 case 일치 (화용론 10·의미론 7)
- 주제 문장의 framing이 후속 case·예시와 의미적으로 일치
- 사례: "그 비용이 다르다" framing → case들이 "Visual Rebuild + 추가 비용"의 추가 부분 변동을 설명 → "거기에 추가로 드는 비용"으로 framing 수정 필요

### G. 산문 우선
- 설명은 산문(문단)으로
- 불릿은 병렬 비교·요약·체크리스트에만

### H. 기술 용어 보존 (의미론 5)
- 정확 용어 ("L2", "필레이트", "텍셀")는 친숙어로 교체 X
- 정확 용어 + 1줄 설명이 정확 용어 삭제보다 우월
