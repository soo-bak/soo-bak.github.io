# 사람이 읽는 자연스러움 점검 (Rewrite by Human)

사람이 직접 읽었을 때 느끼는 **부자연스러움**을 탐지하고 수정합니다. 인과 오류, 논리 모순, 의미 반복, 흐름 단절, 문체 경직 — 기계적 검사로는 잡히지 않지만 독자가 "뭔가 이상하다"고 느끼는 문제를 다룹니다.

**독립 스킬** — 파이프라인에 속하지 않으며, 다른 스킬을 참조하거나 의존하지 않습니다. 단독으로 실행합니다.

## 인자

- `$ARGUMENTS`: 수정할 파일 경로 (선택사항). 없으면 사용자에게 파일 또는 텍스트를 요청합니다.

> **교차 규칙**: `/soobak-writing-review`의 **사용자 문체 규칙**이 이 스킬의 모든 수정에 적용됩니다. 충돌 시 문체 규칙이 우선합니다.

---

## 실행 원칙

**이 스킬이 호출되면 항상 4단계 전체를 빠짐없이 실행한다.** 글이 이미 깔끔해 보이거나 변경할 것이 적어 보여도, 프로세스를 축약하거나 실행이 불필요하다고 판단하지 않는다.

---

## 15가지 검사 패턴

이 스킬의 핵심입니다. 모든 패턴은 하나의 원칙에서 비롯됩니다:

> **사람이 읽었을 때, 논리가 정확하고 흐름이 자연스러워야 한다.**

### A. 인과/논리 오류

#### 패턴 A1. 효과를 이유로 제시

"~하므로/그래서 X가 Y보다 앞에 위치한다" 형태에서, 제시된 이유가 실제로는 결과/효과인 패턴.

**검사 항목:**
- "~하므로", "그래서", "때문에" 뒤에 오는 주장의 근거가 실제로는 결과/부수 효과인가?
- 진짜 이유(구조적 의존성, 입출력 관계, 선행 조건)가 따로 있는가?

**원칙:** 이유는 구조적 의존성(입출력 관계, 선행 조건)이어야 한다. 효과(비용 감소, 성능 향상)는 이유가 아니다.

**수정 방향:** 효과를 걷어내고, 구조적 의존성을 이유로 제시한다.

**사례:**
- Before: "컬링으로 렌더링 대상을 먼저 줄이면 배칭이 처리해야 할 오브젝트 수도 줄어들고, GPU에 제출되는 전체 작업량이 감소합니다. **그래서** 컬링이 배칭보다 앞에 위치합니다."
- After: "컬링이 **무엇을** 렌더링할지 결정하면, 배칭은 그 결과를 받아 드로우콜을 효율적으로 묶어 GPU에 제출합니다."
- 이유: "작업량이 줄어들므로"는 효과. 실제 이유는 "컬링의 출력이 배칭의 입력"이라는 의존성.

#### 패턴 A2. 대상·역할 혼동

비용/문제 자체를 수단으로, 또는 수단을 비용으로 서술하는 패턴.

**검사 항목:**
- "A와 B로 비용을 줄이는" 형태에서, A가 줄여야 할 대상(비용)인데 수단처럼 나열되어 있는가?
- 개념의 역할(원인/결과, 수단/대상, 입력/출력)이 문장 내에서 뒤바뀌어 있는가?

**원칙:** 비용/문제 자체와, 그것을 줄이는 수단/기법을 구분한다.

**수정 방향:** 역할이 뒤바뀐 개념을 정확한 위치에 배치한다.

**사례:**
- Before: "**드로우콜과 배칭**으로 GPU에 렌더링 명령을 제출하는 비용을 줄이는"
- After: "**배칭을 통해** CPU가 GPU에 렌더링 명령을 제출하는 비용을 줄이는"
- 이유: 드로우콜은 줄여야 할 비용이지, 비용을 줄이는 수단이 아님.

#### 패턴 A3. 동일 개념의 모순 서술

같은 개념을 서로 다른 문장에서 상충하게 서술하는 패턴.

**검사 항목:**
- 같은 개념의 위치/역할/특성이 글의 다른 부분에서 모순되는가?
- "~전에"와 "~안에서", "포함"과 "별개" 등 상충하는 서술이 공존하는가?

**원칙:** 하나의 개념은 글 전체에서 일관된 위치/역할을 가진다.

**수정 방향:** 모순되는 서술 중 정확한 것을 택하고, 나머지를 일관되게 수정한다.

**사례:**
- Before: "렌더링 **파이프라인에 진입하기 전에**" + (다른 곳) "렌더링 **파이프라인의 가장 앞 단계에서** 동작합니다"
- After: "**GPU에 제출하기 전에**" (일관된 위치)
- 이유: "진입 전"(밖)과 "가장 앞 단계"(안)는 모순.

### B. 동어 반복

#### 패턴 B1. 의미 반복

한 문단 안에서 같은 의미가 형태만 바뀌어 반복되는 패턴. 세 가지 하위 유형이 있다.

**검사 항목:**
- **(원인-결과 동어)** 원인과 결과가 같은 말인가? ("상태 변경을 줄이면 → 렌더 스테이트 전환 비용이 감소")
- **(표현 교체 반복)** 같은 의미를 다른 표현으로 두 번 이상 쓰는가? ("보이지 않는" ≈ "반영되지 않는")
- **(다단계 반복)** "줄이면 → 줄어들고 → 감소" 식으로 같은 축소 의미를 3번 이상 반복하는가?

**원칙:** 명확한 한 문장이 같은 뜻의 여러 문장보다 낫다. 원인과 결과가 같은 말이면 하나만 남긴다.

**수정 방향:** 가장 명확하고 구체적인 표현 하나를 남기고 나머지를 삭제하거나, 문장 전체를 재구성한다.

**사례 (원인-결과):**
- Before: "상태 변경을 줄이면 CPU가 GPU에 보내는 명령 수가 줄어들고, 렌더 스테이트 전환 비용도 감소합니다"
- After: "SRP Batcher로 렌더 스테이트 전환을 줄이면 CPU 부담이 감소합니다"
- 이유: "상태 변경" = "렌더 스테이트 전환"이므로 원인과 결과가 같은 말.

**사례 (표현 교체):**
- Before: "화면에 **보이지 않는** 오브젝트까지 렌더링한다면 GPU는 결과가 **화면에 반영되지 않는** 작업에"
- After: "화면에 보이지 않는 오브젝트까지 GPU에 제출하면 **결과가 폐기될** 작업에"

**사례 (다단계):**
- Before: "렌더링 대상을 먼저 **줄이면** 배칭이 처리해야 할 오브젝트 수도 **줄어들고**, GPU에 제출되는 전체 작업량이 **감소**합니다"
- After: (문장 전체를 역할 기반으로 재작성)

#### 패턴 B2. 섹션 경계 동어 반복

새 섹션의 첫 문장이 이전 섹션의 결론/내용을 요약 반복하는 패턴.

**검사 항목:**
- 새 섹션의 첫 1~2문장이 이전 섹션에서 이미 설명한 내용을 되풀이하는가?
- "Frustum Culling은 ~하지만" 식으로 이전 섹션을 요약한 뒤 새 주제로 들어가는가?

**원칙:** 새 섹션은 새 정보로 시작한다. 독자는 바로 위 섹션을 방금 읽었다.

**수정 방향:** 이전 섹션 요약을 삭제하고, 새 섹션 고유의 정보(구체적 예시, 핵심 정의)로 바로 시작한다.

**사례:**
- Before: "Frustum Culling은 카메라 시야 밖의 오브젝트를 제거하지만, 시야 안에 있어도 다른 오브젝트에 의해 완전히 가려진 오브젝트까지 걸러내지는 못합니다. 건물 뒤에 있는 가구..."
- After: "건물 뒤에 있는 가구, 벽 뒤의 방, 산 뒤편의 마을처럼 프러스텀 안에 있지만 완전히 가려진 오브젝트는 화면에 한 픽셀도 나타나지 않습니다."

#### 패턴 B3. 소제목·도입문 중복

소제목이 이미 전달한 신호를 도입문이 반복하는 패턴.

**검사 항목:**
- 소제목에 "제약", "장점", "비교" 등이 있는데, 도입문에서 "하지만 제약도 있습니다", "장점은 다음과 같습니다" 등으로 같은 말을 하는가?
- 소제목만으로 독자가 이미 알 수 있는 정보를 도입문이 반복하는가?

**원칙:** 소제목 자체가 전환 신호. 도입문에서 같은 말을 반복하지 않는다.

**수정 방향:** 소제목이 전달하는 정보를 반복하는 도입문을 삭제하고, 바로 첫 번째 구체 정보로 진입한다.

**사례:**
- Before: "### 베이크 방식의 장점과 제약" → "베이크 방식은 런타임 비용이 작습니다." → ... → "하지만 제약도 있습니다."
- After: (장점은 이전 문단에 병합) "### 베이크 방식의 제약" → (바로 첫째 제약으로 진입)

#### 패턴 B4. 정의 분산

같은 개념의 정의가 인접 문장 여러 곳에 나뉘어 있는 패턴.

**검사 항목:**
- 같은 용어/개념이 인접한 2개 이상의 문장에서 각각 부분적으로 정의되는가?
- 두 문장을 합치면 하나의 완결된 정의가 되는가?

**원칙:** 하나의 개념은 한 곳에서 완결적으로 정의한다.

**수정 방향:** 분산된 정의를 하나의 문장/문단으로 병합한다.

**사례:**
- Before: "LOD는 카메라와 오브젝트 사이의 거리에 따라 메쉬의 복잡도를 단계적으로 줄여 정점 처리 비용을 절약하는 기법입니다." + (다음 문단) "컬링이 오브젝트 자체를 렌더링 대상에서 제외하는 것이라면, LOD는 렌더링 대상에 남아 있는 오브젝트의 정점 수를 줄여..."
- After: "컬링이 오브젝트 자체를 렌더링 대상에서 제외한다면, **LOD(Level of Detail)**는 렌더링 대상에 남아 있는 오브젝트의 정점 수를 줄여 GPU 부담을 낮춥니다. 카메라와의 거리에 따라 메쉬의 복잡도를 단계적으로 전환하는 방식이며, ..."

#### 패턴 B5. 같은 예시의 반복 사용

동일한 구체적 예시가 글에서 2회 이상 같은 목적으로 사용되는 패턴.

**검사 항목:**
- 같은 구체적 예시(나무/풀/돌, 건물/가구 등)가 글에서 2회 이상 등장하는가?
- 두 번째 등장에서 새로운 관점이나 정보를 추가하는가, 아니면 같은 원리를 반복 설명하는가?

**원칙:** 예시는 한 번만. 두 번째 등장이 같은 원리의 반복이면 참조로 대체한다.

**수정 방향:** 두 번째 등장을 "앞서 설명한 원리로"로 참조하거나, 다른 예시로 교체한다.

**사례:**
- Before: (2D 아틀라스) "나무, 풀, 돌, 울타리의 텍스처를 하나의 아틀라스에 합치면..." + (3D 아틀라스) "나무, 풀, 돌의 텍스처를 하나의 아틀라스로 합치면..."
- After: (3D) "씬에서 자주 함께 등장하는 오브젝트들의 텍스처를 하나의 아틀라스로 합치면, 앞서 설명한 것과 같은 원리로 배칭이 가능해집니다."

#### 패턴 B6. 시각 요소 직후 산문 재나열

표·다이어그램 직후의 산문이 시각 요소의 내용을 그대로 반복 나열하는 패턴.

**검사 항목:**
- 표·다이어그램 직후 산문이 시각 요소가 보여주는 항목을 순서대로 다시 나열하는가?
- 산문에 시각 요소에 없는 새로운 정보(맥락, 해석, 예외)가 포함되어 있는가?

**원칙:** 산문은 시각 요소에 없는 새로운 맥락·해석만 추가한다. 시각 요소가 보여주는 목록을 산문에서 다시 펼치지 않는다.

**수정 방향:** 시각 요소와 겹치는 나열을 삭제하고, 시각 요소가 전달하지 못하는 요약·결론·맥락만 남긴다.

**사례:**
- Before: (비교 표 직후) "이 기법들은 서로 배타적이지 않고 단계적으로 적용됩니다. Frustum Culling이 프러스텀 밖의 오브젝트를 제거하고, Occlusion Culling이 가려진 오브젝트를 추가로 제거합니다. 레이어별 컬링 거리가 먼 거리의 소형 오브젝트를 걸러내고, LOD가 남은 오브젝트의 삼각형 수를 줄입니다. 텍스처 아틀라스는 머티리얼을 통일하여 배칭을 가능하게 합니다."
- After: "이 기법들은 서로 배타적이지 않고 단계적으로 적용되어, 최종적으로 GPU에 제출되는 작업량을 최소화합니다."

**주의:** 개념 다이어그램 내부의 "→" 결론 라인은 중복이 아닙니다. 다이어그램을 자체 완결적으로 만드는 요소이므로 유지합니다.

### C. 흐름 단절

#### 패턴 C1. 목적 없이 정의 등장

용어 정의가 "왜 필요한가"에 대한 맥락 없이 등장하는 패턴.

**검사 항목:**
- 용어 정의 직전에, 그 용어가 해결하는 문제나 목적이 제시되어 있는가?
- 독자가 "이게 왜 필요하지?"라고 느낄 수 있는가?

**원칙:** 정의 전에 목적(왜 이것이 필요한가, 어떤 문제를 푸는가)이 먼저 깔려야 한다.

**수정 방향:** 정의에 목적절("~를 위해", "~을 해결하기 위해")을 추가하거나, 정의 전에 문제 상황을 먼저 제시한다.

**사례:**
- Before: "GPU에 제출하기 전에 화면에 보이지 않는 오브젝트를 걸러내는 과정이 **컬링(Culling)**입니다."
- After: "**컬링(Culling)**은 **불필요한 드로우콜을 없애기 위해**, GPU에 제출하기 전에 화면에 보이지 않는 오브젝트를 걸러내는 과정입니다."
- 사용자 피드백: "의 목적이 뭔지 모르니까 급작스러운거같아 연결이 안되고"

#### 패턴 C2. 앞뒤 연결 없는 독립 문장

문장이 앞뒤 맥락과 논리적 연결 없이 떠 있는 패턴.

**검사 항목:**
- 문장을 삭제해도 앞뒤 흐름이 변하지 않는가?
- 앞 문장과의 인과/전제/결과 관계가 보이지 않는가?
- 단독 사실만 선언하는 짧은 문장이 맥락 없이 놓여 있는가?

**원칙:** 모든 문장은 앞 문장의 결과이거나 다음 문장의 전제. 연결이 없으면 맥락에 녹이거나 삭제한다.

**수정 방향:** 독립 문장의 정보를 인접 문장에 자연스럽게 녹이거나, 앞뒤와의 논리적 연결을 명시한다.

**사례:**
- Before: "컬링은 CPU에서 수행됩니다. 씬에 오브젝트가 1,000개 있어도 카메라에 보이는 100개만 GPU에 제출하면..."
- After: "씬에 오브젝트가 1,000개 있더라도 **CPU에서 컬링을 거치면** 카메라에 보이는 100개만 GPU에 제출되고..."
- 사용자 피드백: "두 문장이 관련이 없는것같은데"

#### 패턴 C3. 불필요한 회귀 표현

"앞서", "앞의 절들에서 다룬", "앞에서 ~라고 했습니다" 같은 되짚기 표현 패턴.

**검사 항목:**
- "앞서", "위에서", "앞의 절들에서 다룬", "앞에서 ~라고 했습니다", "~에서 언급한 것처럼" 등의 회귀 표현이 있는가?
- 회귀 없이 직접 대조하거나 새 정보로 시작할 수 있는가?

**원칙:** 회귀 없이 직접 대조·연결한다. 독자는 바로 위를 방금 읽었다.

**수정 방향:** 회귀 표현을 삭제하고, 대조 구문("A가 ~이라면, B는 ~이다")이나 직접 서술로 대체한다.

**사례:**
- Before: "**앞의 절들에서 다룬** 컬링과 LOD는 렌더링 대상 오브젝트의 수나 복잡도를 줄이는 방법입니다."
- After: "컬링과 LOD**가** 렌더링 대상의 수나 복잡도를 줄이는 방법**이라면**, 텍스처 아틀라스는..."

**사례 2:**
- Before: "**앞에서 CPU는 레이턴시를 줄이기 위해 코어 하나를 정교하게 만든다고 했습니다.** GPU는 정반대입니다."
- After: "**CPU가 코어 하나의 레이턴시를 줄이는 데 트랜지스터를 투자했다면**, GPU는 그 트랜지스터로 단순한 코어를 수천 개 배치합니다."

### D. 문체 경직

#### 패턴 D1. 교과서식 나열 도입문

"~에는 A, B, C가 있습니다", "세 가지 옵션이 있습니다" 형태의 목록 예고 패턴.

**검사 항목:**
- "~에는 N가지가 있습니다", "N개의 옵션이 있습니다", "다음과 같은 방법들이 있습니다" 형태의 예고문이 있는가?
- 예고문 없이 바로 첫 항목 설명으로 진입해도 독자가 이해하는가?

**원칙:** 목록 예고 없이 바로 항목 설명으로 진입한다.

**수정 방향:** 예고문을 삭제하고, 첫 번째 항목의 설명으로 바로 시작한다.

**사례:**
- Before: "카메라에는 **세 가지 설정이 있습니다.**" + FOV 설명 + Near 설명 + Far 설명
- After: (삭제) "**시야각(Field of View)**은 카메라가 한 번에 볼 수 있는 각도 범위를 결정합니다." 로 바로 진입

#### 패턴 D2. 불필요한 나열/펼침

상위 개념으로 요약 가능한 내용을 하나하나 펼쳐 쓰는 패턴.

**검사 항목:**
- 나열된 항목들이 모두 하나의 상위 개념에 속하는가?
- 개별 항목의 구분이 맥락상 중요한가, 아니면 "전체"로 요약해도 의미가 보존되는가?

**원칙:** 개별 항목의 구분이 중요하지 않으면 상위 개념으로 요약한다.

**수정 방향:** 하위 항목 나열을 상위 개념 한 단어로 대체한다.

**사례:**
- Before: "나머지 900개에 대한 **드로우콜 제출, 정점 처리, 프래그먼트 처리**가 모두 생략됩니다"
- After: "나머지 900개는 **렌더링 자체가** 생략됩니다"
- 사용자 피드백: "가 여전히 부자연스럽게 느껴져"

#### 패턴 D3. 이질적 항목의 모호한 병합

역할이 다른 항목들을 "이와 함께", "그 외에도" 등으로 모호하게 이어붙이는 패턴.

**검사 항목:**
- "이와 함께", "그 외에도", "또한", "아울러" 등으로 이어진 항목들이 실제로 같은 역할/범주인가?
- 역할이 다른 항목이 구분 없이 나열되어 있는가?

**원칙:** 역할이 다르면 역할 기준으로 구분한다.

**수정 방향:** 역할 기준으로 항목을 분리하고, 각 역할을 명시한다.

**사례:**
- Before: "Unity에서 사용되는 컬링 기법에는 Frustum Culling, Occlusion Culling, 레이어별 컬링 거리가 있습니다. **이와 함께** LOD와 텍스처 아틀라스도 렌더링 비용 절감에 기여합니다."
- After: "Unity에서는 Frustum Culling, Occlusion Culling, 레이어별 컬링 거리**로 렌더링 대상을 줄이고**, LOD와 텍스처 아틀라스**로 남은 오브젝트의 렌더링 비용을 낮춥니다**."

## 작업 프로세스

### Phase 1: 글 전체 파악

1. `$ARGUMENTS`가 있으면 해당 파일을 `Read`로 전체 읽기. 없으면 사용자에게 파일 경로 요청.
2. `Grep`으로 `##`/`###` 제목을 추출하여 **섹션 지도** 작성.
3. 각 섹션의 역할 식별: 도입, 개념 소개, 상세 설명, 예시, 비교, 마무리.
4. 시각 요소(표, SVG/ASCII 다이어그램, 코드 블록) 위치와 내용을 목록화.

### Phase 2: 15가지 패턴 검사

각 패턴의 검사 항목을 순서대로 적용하고, 발견된 문제를 기록합니다.

기록 형식:
```
| 패턴 | 위치(line) | 문제 내용 | 수정 방향 |
|------|-----------|----------|----------|
| A1   | L42       | "줄어들므로 앞에 위치" | 효과→구조적 의존성 |
| B1   | L15       | "상태 변경 = 렌더 스테이트 전환" 원인-결과 동어 | 하나로 축약 |
| C3   | L120      | "앞의 절들에서 다룬" | 대조 구문으로 교체 |
```

**이 단계에서는 Edit를 실행하지 않습니다.** 전체 검사를 먼저 완료합니다.

### Phase 3: 수정 실행

Phase 2의 기록을 바탕으로 `Edit`를 적용합니다.

**수정 순서:**
1. **논리 수정** (A1~A3) — 인과/논리 오류를 먼저 바로잡는다. 논리가 틀리면 다른 수정이 무의미.
2. **반복 제거** (B1~B6) — 의미 반복, 정의 분산, 예시 반복 등을 정리.
3. **흐름 연결** (C1~C3) — 독립 문장을 맥락에 녹이고, 회귀 표현을 제거.
4. **문체 정리** (D1~D3) — 교과서식 나열, 불필요 펼침, 모호한 병합을 수정.

**수정 시 절대 규칙:**
- **정보 손실 금지** — 삭제·축약 시 해당 정보가 글의 다른 위치(또는 시각 요소)에 존재하는지 반드시 확인. 글 전체에서 사라지는 정보가 있으면 안 됨.
- **기술 용어 보존** — 수정 과정에서 기술 용어(필레이트, 텍셀, 드로우콜 등)가 탈락하지 않았는지 before/after를 대조.
- **구체적 수치 보존** — 특정 숫자, 메커니즘, 단계별 설명이 수정 과정에서 사라지지 않았는지 확인.

### Phase 4: 자기 검증 + 보고

1. 수정된 텍스트를 `Read`로 다시 읽는다.
2. 15가지 패턴을 재검사한다. 위반이 남아 있으면 추가 `Edit`로 교정한다.
3. **정보 손실 점검** — Phase 2에서 기록한 각 수정 항목에 대해, 해당 정보가 글의 어딘가에 존재하는지 확인한다.
4. 결과를 아래 형식으로 보고한다.

## 자기 검증 체크리스트

모든 Edit 적용 후, 수정된 텍스트를 Read로 다시 읽고 아래 항목을 검증합니다:

1. **A1~A3 잔존** — 효과를 이유로 제시하는 곳, 대상·역할이 혼동된 곳, 모순 서술이 남아 있지 않은가?
2. **B1 잔존** — 같은 의미가 표현만 바뀌어 반복되는 곳이 남아 있지 않은가?
3. **B2~B3 잔존** — 섹션 경계 반복, 소제목·도입문 중복이 남아 있지 않은가?
4. **B4 잔존** — 같은 개념의 정의가 여러 곳에 분산되어 있지 않은가?
5. **B5~B6 잔존** — 같은 예시 반복, 시각 요소 직후 재나열이 남아 있지 않은가?
6. **C1~C3 잔존** — 목적 없이 정의가 등장하는 곳, 독립 문장, 회귀 표현이 남아 있지 않은가?
7. **D1~D3 잔존** — 교과서식 나열, 불필요 펼침, 모호한 병합이 남아 있지 않은가?
8. **정보 손실** — 수정 전에 있던 기술 용어·수치·메커니즘이 글 전체에서 사라지지 않았는가?
9. **수정으로 인한 새 문제** — 수정으로 인해 앞뒤 문맥이 끊기거나, 참조가 깨진 곳이 없는가?
10. **전체 통독** — 수정된 섹션 전체를 처음부터 끝까지 하나의 글로 읽었을 때, 흐름이 자연스러운가?
