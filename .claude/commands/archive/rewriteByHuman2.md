# 구조·맥락·정확성 점검 (Rewrite by Human 2)

`rewriteByHuman`이 문장 단위의 자연스러움(인과 오류, 의미 반복, 흐름 단절, 문체 경직)을 다룬다면, 이 스킬은 **문단 간 연결, 시리즈 간 일관성, 기술적 범위·정확성, 문장 구조** 문제를 다룹니다. GameLoop, RenderingFoundation, GPUArchitecture, UnityPipeline 시리즈 리뷰 세션에서 반복적으로 발견·수정된 패턴을 체계화한 것입니다.

**독립 스킬** — 파이프라인에 속하지 않으며, 단독으로 실행합니다.

## 인자

- `$ARGUMENTS`: 수정할 파일 경로 (선택사항). 없으면 사용자에게 파일 경로를 요청합니다.
- 시리즈 글인 경우, 이전/이후 파트 파일도 함께 읽어 교차 점검합니다.

> **교차 규칙**: `/soobak-writing-review`의 **사용자 문체 규칙**이 이 스킬의 모든 수정에 적용됩니다. 충돌 시 문체 규칙이 우선합니다.

---

## 실행 원칙

**이 스킬이 호출되면 항상 5단계 전체를 빠짐없이 실행한다.** 글이 이미 깔끔해 보이거나 변경할 것이 적어 보여도, 프로세스를 축약하거나 실행이 불필요하다고 판단하지 않는다.

---

## 20가지 검사 패턴

모든 패턴은 하나의 원칙에서 비롯됩니다:

> **문단과 문단, 글과 글 사이의 연결이 정확하고 자연스러워야 한다.**

---

### E. 구조적 연결 결함

#### 패턴 E1. Recap→Definition 동기 부재

이전 글/섹션을 되짚는 문장 직후에, "왜 이것이 문제인가"라는 동기 없이 곧바로 새 개념의 정의가 등장하는 패턴.

**검사 항목:**
- 이전 글/섹션 되짚기(메타 서술) 바로 다음에 용어 정의가 오는가?
- 되짚기와 정의 사이에 "그런데 ~에 비용이 발생한다", "하지만 ~가 문제가 된다" 같은 동기 문장이 있는가?

**원칙:** 되짚기(recap) → 동기(motivation) → 정의(definition) 순서를 지킨다. 동기가 빠지면 정의가 급작스럽다.

**수정 방향:** 되짚기와 정의 사이에, 해당 개념이 왜 중요한지를 밝히는 동기 문장을 삽입한다.

**사례 (UnityPipeline-2):**
- Before: "[Part 1]에서는 렌더 파이프라인이 씬을 렌더링하기 위해 GPU에 명령을 보내는 구조를 살펴보았습니다. CPU는 오브젝트를 그릴 때마다 GPU에 렌더링 명령을 보내는데, 이를 **드로우콜**이라 합니다."
- After: "[Part 1]에서는 ... 구조를 살펴보았습니다. **그런데 이 명령을 보내는 과정 자체에 비용이 발생합니다.** CPU는 오브젝트를 그릴 때마다 ..."
- 이유: "구조를 살펴보았습니다"(메타) → "이를 드로우콜이라 합니다"(정의) 사이에 "왜 비용이 문제인가"라는 동기가 빠져 있었음.

#### 패턴 E2. 단문 나열 (위장된 불릿 리스트)

단일 문장 문단이 `<br>` 태그와 함께 글머리 기호 없이 나열되는 패턴. 산문을 지향하는 글에서 흐름이 끊긴다.

**검사 항목:**
- 연속된 3개 이상의 문단이 각각 1~2문장인가?
- 문단 사이에 `<br>` 태그가 있는가?
- 해당 문단들이 하나의 주제에 대한 병렬 서술인가?

**원칙:** 같은 주제의 병렬 서술은 하나의 문단으로 합친다. 산문 흐름을 유지한다.

**수정 방향:** 병렬 단문들을 접속사("~고", "~며", "또한")로 연결하여 하나의 문단으로 결합하고, 불필요한 `<br>` 태그를 제거한다.

**사례 (UnityPipeline-2 장점 섹션):**
- Before: 4개 단문 문단 각각 1문장 + `<br>` — "SetPass Call이 감소합니다." / "메모리 증가가 없습니다." / "런타임 CPU 비용이 추가되지 않습니다." / "움직이는 오브젝트에도 동작합니다."
- After: "SetPass Call이 감소합니다. 메쉬를 합치지 않기 때문에 **메모리 증가가 없고**, Dynamic Batching처럼 매 프레임 정점을 변환하는 작업도 필요 없어 **런타임 CPU 비용이 추가되지 않습니다**. 또한 ... **움직이는 오브젝트에도 동작**합니다."

**사례 (GPUArchitecture-2 볼드 레이블):**
- Before: `<br>` + `**FlexRender**` / `<br>` + `**LRZ**` 독립 레이블
- After: 인트로에서 FlexRender로 자연 연결, LRZ는 흐르는 산문으로 통합

#### 패턴 E3. 불필요한 문단 분리

논리적으로 이어지는 문장이 별도 문단으로 나뉘어 흐름이 끊기는 패턴.

**검사 항목:**
- 연속된 두 문단이 한 문단에서 자연스럽게 이어지는 내용인가?
- 문단 분리가 주제 전환 시가 아닌데 분리되어 있는가?

**원칙:** 같은 논리 흐름 안의 문장은 같은 문단에 둔다. 문단 분리는 주제 전환 시에만.

**수정 방향:** 논리적으로 연결된 문단을 합치고, 필요시 접속사로 연결한다.

**사례 (UnityPipeline-2):**
- Before: "이를 **배칭**이라 합니다. Unity는 네 가지 배칭 기법을 제공합니다." + (별도 문단) "이 글에서는 드로우콜의 비용이 어디서 발생하는지 분석한 뒤, ..."
- After: "... 네 가지 배칭 기법을 제공하며, 이 글에서는 드로우콜의 비용이 ... 살펴봅니다."

**사례 (RenderingFoundation-3):**
- Before: 블렌딩 정의(문단 1) + 블렌딩 적용(문단 2) → 분리되어 뜬금없음
- After: 정의 → 불투명 → 반투명을 하나의 문단으로 합쳐 블렌딩의 맥락이 연속

#### 패턴 E4. 맥락 전환 없는 주제 전환

이전 주제를 깊이 있게 다룬 직후, 전환 문장 없이 새 주제로 넘어가는 패턴.

**검사 항목:**
- 이전 섹션/문단의 마지막 문장과 다음 문단의 첫 문장 사이에 주제가 바뀌는가?
- 주제 전환을 알리는 전환 문장("이러한 ~ 외에도", "이제 ~를 살펴보면")이 있는가?

**원칙:** 주제가 바뀔 때 독자에게 전환을 알려야 한다.

**수정 방향:** 전환 문장을 삽입하거나, 이전 주제의 마무리 문장에 다음 주제로의 연결을 추가한다.

**사례 (RenderingFoundation-3):**
- Before: 좌표 변환을 깊이 있게 다룬 직후 맥락 전환 없이 프래그먼트 셰이더 설명 시작
- After: 세 단계 요약(텍스처 샘플링 → 조명 계산 → 최종 색상 결합)을 한 문단으로 미리 보여준 뒤 상세 설명 진입

#### 패턴 E5. 문단 밀도 과부하

한 문단에서 서로 다른 역할의 내용(정의, 목차 나열, 시리즈 연결 등)을 동시에 수행하여 밀도가 과도한 패턴.

**검사 항목:**
- 한 문단에서 3가지 이상의 서로 다른 역할을 수행하는가?
- 문단 안에서 논리적 단절이 느껴지는가?

**원칙:** 한 문단은 하나의 역할. 역할이 다르면 문단을 나눈다.

**수정 방향:** 역할별로 문단을 분리한다.

**사례 (RenderingFoundation-3):**
- Before: 한 문단에 (1) 머티리얼 정의 + (2) 글의 목차 나열 + (3) 시리즈 연결 → 밀도 과부하
- After: 머티리얼 정의 → 시리즈 연결(세 가지 핵심 데이터)을 먼저 한 호흡에 연결, 목차 나열은 별도 문단으로 분리

### F. 시리즈 간 일관성

#### 패턴 F1. 시리즈 간 용어 재정의

이전 파트에서 이미 정의한 용어를 현재 파트에서 다시 **`이를 ~이라 합니다`** 형식으로 정의하는 패턴.

**검사 항목:**
- 글에 **볼드 + 괄호** 형식(`**용어(English)**`)으로 정의되는 용어가 있는가?
- 해당 용어가 시리즈의 이전 파트에서 이미 정의되었는가?
- 현재 파트에서 해당 용어를 더 깊이 설명하는 전용 섹션이 따로 있는가?

**원칙:** 이전 파트에서 정의된 용어는 이미 알려진 것으로 취급한다. 도입부에서 재정의하지 않고, 상세 설명이 필요하면 전용 섹션에서 확장한다.

**수정 방향:** 재정의 형식("이를 **X**라 합니다")을 제거하고, 기존 용어로 자연스럽게 사용한다.

**사례 (UnityPipeline-2):**
- Before: "CPU는 오브젝트를 그릴 때마다 GPU에 렌더링 명령을 보내는데, 이를 **드로우콜(Draw Call)**이라 합니다." (Part 1 line 24에서 이미 정의됨)
- After: 도입부에서 재정의 제거, 바로 아래 "## 드로우콜의 정의" 섹션이 상세 설명 담당

#### 패턴 F2. 시리즈 간 사실관계 불일치

현재 파트의 서술이 이전/이후 파트의 서술과 모순되는 패턴.

**검사 항목:**
- 현재 글에서 단정적으로 서술하는 사실이, 이전/이후 파트에서 조건부·예외 포함으로 서술되어 있지 않은가?
- 같은 개념에 대해 파트마다 범위·전제가 다르게 기술되어 있지 않은가?

**원칙:** 시리즈 전체에서 같은 개념은 일관된 범위와 전제로 기술한다.

**수정 방향:** 이전/이후 파트를 읽고, 범위·조건·예외가 일치하도록 수정한다.

**사례 (GPUArchitecture-2):**
- Before (Part 2): "Unity의 Opaque 패스는 불투명 오브젝트를 앞에서 뒤로 정렬합니다." (무조건적 서술)
- Part 1의 실제 서술: "`OpaqueSortMode.Default`는 GPU에 따라 다르며, PowerVR/Apple GPU에서는 front-to-back을 적용하지 않습니다."
- After: Mali/Adreno 섹션에만 front-to-back 언급, Apple 섹션에는 "정렬 순서가 성능에 미치는 영향이 크지 않습니다" 명시

#### 패턴 F3. 다른 파트 범위 침범

현재 파트의 서술이 다른 파트에서 다루는 주제를 마치 이 글에서 다룬 것처럼 참조하는 패턴.

**검사 항목:**
- "앞에서 다룬 X"라고 쓰는데, X가 실제로는 다른 파트의 주제인가?
- 현재 파트에서 다루지 않은 개념을 기정사실처럼 참조하는가?

**원칙:** 참조할 때는 어디서 다루는지 정확히 명시한다.

**수정 방향:** 잘못된 참조를 정확한 파트 번호로 교체하거나, 해당 참조를 제거한다.

**사례 (UnityPipeline-2):**
- Before: Part 1을 되짚으며 "컬링과 정렬을 거쳐" 사용 → 컬링은 Part 3의 주제
- After: "컬링" 참조 제거, Part 1의 실제 범위(렌더 파이프라인의 GPU 명령 제출 구조)에 맞게 수정

### G. 범위·정확성 오류

#### 패턴 G1. 범용 개념의 엔진 귀속

그래픽스 API 전반에서 사용되는 일반 개념을 특정 엔진(Unity, Unreal 등)의 고유 기능처럼 서술하는 패턴.

**검사 항목:**
- "Unity는 ~를 제공합니다", "Unity의 ~기법" 형태에서, 해당 개념이 실제로 Unity 고유인가?
- 배칭, 컬링, LOD, 인스턴싱 등 그래픽스 일반 개념을 엔진 귀속으로 서술하고 있는가?

**원칙:** 일반 그래픽스 개념은 일반 개념으로 먼저 정의하고, 엔진의 구체적 구현은 별도로 구분한다.

**수정 방향:** 일반 개념 정의 → 엔진 구현 순서로 재구성한다.

**사례 (UnityPipeline-2):**
- Before: "이 비용을 줄이기 위해 Unity는 여러 배칭(Batching) 기법을 제공합니다."
- After: "이 비용을 줄이기 위해 여러 드로우콜을 묶거나 상태 변경을 줄이는 접근을 사용하는데, 이를 **배칭(Batching)**이라 합니다. Unity는 ... 네 가지 배칭 기법을 제공하며, ..."
- 이유: 배칭은 DirectX, OpenGL, Vulkan 등 그래픽스 API 전반의 일반 개념.

#### 패턴 G2. 불필요한 플랫폼 한정

일반적으로 적용되는 문제/현상을 특정 플랫폼(모바일, PC 등)에 한정하여 서술하는 패턴.

**검사 항목:**
- "모바일에서는", "PC에서는" 같은 한정 표현이 있는가?
- 해당 문제가 실제로 해당 플랫폼에만 해당하는가, 아니면 범용적인가?

**원칙:** 플랫폼에 관계없이 적용되는 현상은 한정 없이 서술한다. 플랫폼별 차이가 있을 때만 한정한다.

**수정 방향:** 불필요한 플랫폼 한정 표현을 제거한다.

**사례 (UnityPipeline-2):**
- Before: "모바일에서는 드로우콜을 준비하고 전달하는 CPU 측 비용이 병목이 되는 경우가 많아 ..."
- After: "드로우콜을 준비하고 전달하는 CPU 측 비용이 병목이 되어 ..."
- 이유: 드로우콜 비용은 플랫폼 공통. 모바일에서 더 심하다는 것은 별도 섹션(모바일 드로우콜 예산)에서 다룸.

#### 패턴 G3. 조건부 사실의 무조건적 서술

조건이나 예외가 있는 사실을 무조건적으로 단정하는 패턴.

**검사 항목:**
- "A는 B입니다", "A이므로 B입니다" 형태에서, 실제로는 조건부(일부 GPU, 특정 설정 등)인가?
- 같은 글/시리즈의 다른 곳에서 예외를 명시하고 있는가?

**원칙:** 조건부인 사실은 조건을 명시한다. "대부분의 경우", "~에서는", "기본적으로" 등을 사용한다.

**수정 방향:** 무조건적 서술에 조건/예외를 추가한다.

**사례 (GPUArchitecture-2 / UnityPipeline-2):**
- Before: "정렬 기준이 머티리얼입니다." (무조건적)
- After: "정렬 기준 **중 하나가** 머티리얼입니다."

- Before: "Unity의 Opaque 패스는 불투명 오브젝트를 앞에서 뒤로 정렬합니다." (모든 GPU)
- After: "**대부분의 GPU에서** 앞에서 뒤로 정렬합니다." (PowerVR/Apple 예외)

#### 패턴 G4. 메커니즘 오귀속

결과나 효과를 잘못된 메커니즘에 귀속시키는 패턴.

**검사 항목:**
- "A가 B를 돕기 위해서입니다" 형태에서, A가 실제로 B에 기여하는가?
- 같은 결과를 만드는 다른 메커니즘이 있는데 혼동하고 있는가?

**원칙:** 효과를 정확한 메커니즘에 귀속시킨다.

**수정 방향:** 정확한 메커니즘을 파악하여 귀속을 교정한다.

**사례 (GPUArchitecture-2 Adreno 섹션):**
- Before: "가까운 오브젝트의 깊이가 먼저 기록되면, LRZ 버퍼에 더 가까운 깊이가 기록되어 셰이딩 효율이 높아집니다."
- After: "가까운 오브젝트의 깊이가 먼저 기록되면, **타일별 렌더링 단계에서 Early-Z가 뒤쪽 프래그먼트를 더 많이 거부할 수 있어** 셰이딩 효율이 높아집니다."
- 이유: LRZ 버퍼는 Binning 단계에서 모든 삼각형이 처리된 후 만들어지므로 순서에 무관(order-independent). front-to-back이 돕는 것은 LRZ가 아니라 per-tile Early-Z.

#### 패턴 G5. 역할/주체 혼동

개념의 역할(정의자/실행자, 데이터/프로그램, 수단/대상)을 혼동하여 서술하는 패턴.

**검사 항목:**
- "A가 B를 정의/결정한다" 형태에서, 실제 주체가 A가 아닌 다른 것인가?
- 데이터와 프로그램, 수단과 대상의 구분이 정확한가?

**원칙:** 각 개념의 역할(정의자/피정의자, 실행 주체/데이터, 수단/대상)을 정확히 구분한다.

**수정 방향:** 역할이 혼동된 주체를 정확한 위치에 배치한다.

**사례 (RenderingFoundation-3):**
- Before: "**머티리얼(Material)**이 바로 이 렌더링 규칙을 정의합니다."
- After: "이 렌더링 규칙을 정의하는 프로그램이 **셰이더(Shader)**이고, 셰이더가 요구하는 구체적인 파라미터 값들을 하나로 묶어 저장하는 데이터 묶음이 **머티리얼(Material)**입니다."
- 이유: 렌더링 규칙을 정의하는 주체는 셰이더. 머티리얼은 파라미터 값을 담는 데이터.

#### 패턴 G6. 억지 1:1 매핑

실제로는 다대다(N:N) 관계인 것을 억지로 1:1로 대응시키는 패턴.

**검사 항목:**
- "A는 B에, C는 D에 대응한다" 형태에서, 실제로 A가 D에도 영향을 주는가?
- 깔끔한 분류를 위해 사실관계를 왜곡하고 있는가?

**원칙:** 대응 관계가 정확하지 않으면 억지로 나누지 않는다.

**수정 방향:** 부정확한 1:1 매핑을 제거하고, 공통 원리로 묶어서 서술한다.

**사례 (GameLoop-2):**
- Before: "필레이트 제한 → 셰이더 복잡도/오버드로우, 대역폭 제한 → 텍스처/후처리" (1:1 매핑)
- After: 두 제약을 묶어 언급하고 관리해야 할 요소들을 나열
- 이유: 셰이더 복잡도는 필레이트에도 영향을 주고, 후처리는 대역폭에도 영향. 깔끔하게 분리되지 않는 것을 억지로 나누면 잘못된 인상.

### H. 문장 수준 문제

#### 패턴 H1. 부정확한 인과

"~하므로", "~때문에" 뒤의 결론이 제시된 이유에서 논리적으로 따라오지 않는 패턴. (rewriteByHuman의 A1 "효과를 이유로 제시"와 구분: A1은 효과↔원인 혼동, H1은 인과 자체가 성립하지 않는 경우.)

**검사 항목:**
- "~하므로", "~때문에", "~이기에" 뒤의 결론이, 제시된 전제에서 필연적으로 도출되는가?
- 전제가 참이라도 결론이 다를 수 있는가?

**원칙:** 인과 관계는 전제에서 결론이 필연적으로 도출될 때만 "~하므로"를 사용한다.

**수정 방향:** 부정확한 인과를 정확한 인과로 교체한다.

**사례 (UnityPipeline-2):**
- Before: "GPU는 스스로 그릴 대상을 결정하지 못하므로 오브젝트마다 별도의 드로우콜이 필요합니다."
- After: "각 오브젝트는 서로 다른 메쉬와 머티리얼을 가지므로, 기본적으로 오브젝트마다 별도의 드로우콜이 필요합니다."
- 이유: "GPU가 수동적"에서 "오브젝트마다 별도"가 필연적으로 도출되지 않음 — 배칭은 하나의 명령으로 여러 오브젝트를 그림.

#### 패턴 H2. 문장 과적재

하나의 문장에 나열 항목, 수식어, 부연 설명이 과도하게 몰려 가독성이 떨어지는 패턴.

**검사 항목:**
- 하나의 문장 안에 고유명사가 4개 이상 나열되는가?
- 문장의 주절과 목적절 사이에 나열이 끼어들어 구조가 불명확한가?

**원칙:** 나열이 문장 구조를 압도하면 분리한다.

**수정 방향:** 나열 항목을 별도 문장으로 분리하거나, 문장 구조를 재배치한다.

**사례 (UnityPipeline-2):**
- Before: "이 글에서는 드로우콜의 비용이 정확히 어디서 발생하는지 분석한 뒤, Unity가 제공하는 Static Batching, Dynamic Batching, SRP Batcher, GPU Instancing 네 가지 기법의 원리와 트레이드오프를 살펴봅니다."
- After: "Unity는 Static Batching, Dynamic Batching, SRP Batcher, GPU Instancing 네 가지 배칭 기법을 제공하며, 이 글에서는 드로우콜의 비용이 어디서 발생하는지 분석한 뒤 각 기법의 원리와 트레이드오프를 살펴봅니다."

#### 패턴 H3. 도입부↔하위 내용 중복

상위 섹션 도입부에서 일반적으로 서술한 내용을, 하위 소제목이 거의 같은 말로 반복하는 패턴.

**검사 항목:**
- 상위 섹션(`##`)의 도입부와 하위 소제목(`###`) 첫 문장이 같은 내용을 다른 표현으로 반복하는가?
- 같은 구체적 예시(나무, 풀, 돌 등)가 도입부와 하위 섹션 모두에 등장하는가?

**원칙:** 도입부에서 일반 원리를 설명했으면, 하위 섹션은 도입부에 없는 새 정보로 시작한다.

**수정 방향:** 하위 섹션에서 도입부와 중복되는 첫 문장을 제거하고, 새 정보로 바로 진입한다.

**사례 (UnityPipeline-2):**
- Before (도입부): "나무, 풀, 돌, 총알 등 동일한 메쉬가 수백~수천 개 반복되는 상황에 적합합니다."
- Before (### 적합한 상황): "GPU Instancing은 **동일한 메쉬가 수백~수천 개 반복**되는 상황에서 효과가 큽니다. 숲의 나무, 풀, 돌처럼..."
- After (### 적합한 상황): "숲의 나무, 풀, 돌처럼 같은 에셋이 대량으로 배치되는 환경이 대표적입니다. 전략 게임에서..."

#### 패턴 H4. 용어 사용 후 뒤늦은 정의

용어를 먼저 사용한 뒤 나중에 정의하여 순서가 역전된 패턴.

**검사 항목:**
- 용어가 정의 없이 먼저 사용된 뒤, 이후 문장에서 뒤늦게 정의되는가?
- 독자가 처음 등장 시점에 의미를 알 수 없는가?

**원칙:** 용어는 첫 등장 시점에 정의한다. 사용 → 정의 순서는 역전.

**수정 방향:** 정의를 첫 등장 위치로 앞당기거나, 첫 등장 시 간략 부연을 추가한다.

**사례 (GameLoop-2):**
- Before: "콜드 스타트"를 먼저 사용 → 뒤늦게 정의
- After: 콜드 스타트 정의를 첫 등장 위치에 배치

#### 패턴 H5. 부적절한 단어 선정

문맥에 맞지 않는 어휘를 사용하여 의미가 왜곡되거나 부자연스러운 패턴.

**검사 항목:**
- 선택한 단어의 사전적 의미가 문맥에서 전달하려는 의미와 정확히 일치하는가?
- 더 정확한 대체 단어가 있는가?

**원칙:** 문맥에 가장 정확한 단어를 선택한다.

**수정 방향:** 부적절한 단어를 문맥에 맞는 정확한 단어로 교체한다.

**사례 (UnityPipeline-2):**
- Before: "카메라에 **포착**되고" — 포착은 "잡아냄"의 의미로, 카메라가 능동적으로 무언가를 잡는 느낌
- After: Part 1의 실제 범위에 맞게 "렌더 파이프라인이 씬을 렌더링하기 위해 GPU에 명령을 보내는 구조"로 수정

#### 패턴 H6. 암묵적 전제 생략

여러 논리적 단계가 전제 설명 없이 압축되어, 독자가 논리를 따라가지 못하는 패턴.

**검사 항목:**
- 한 문단에 3단계 이상의 논리가 암묵적 전제 없이 압축되어 있는가?
- 독자가 각 단계 사이의 연결을 스스로 추론해야 하는가?

**원칙:** 각 논리 단계의 전제를 명시한다. 독자에게 추론을 맡기지 않는다.

**수정 방향:** 암묵적 전제를 명시적 문장으로 풀어쓴다.

**사례 (GPUArchitecture-2 Mali 섹션):**
- Before: 한 문단에 FPK의 한계 → Early-Z의 역할 → front-to-back 정렬 이유가 암묵적 전제 없이 압축
- After: 각 단계를 분리하고, "이런 경우를 줄이려면 Early-Z가 셰이딩 전에 최대한 많이 걸러내야 하며"와 같이 전제를 명시

## 작업 프로세스

### Phase 1: 글 전체 파악 + 시리즈 맥락 확인

1. `$ARGUMENTS`가 있으면 해당 파일을 `Read`로 전체 읽기. 없으면 사용자에게 파일 경로 요청.
2. `Grep`으로 `##`/`###` 제목을 추출하여 **섹션 지도** 작성.
3. **시리즈 파일 확인**: 파일명에서 시리즈명과 파트 번호를 식별. 이전/이후 파트가 있으면 `Read`로 읽어 교차 점검 준비.
4. 이전 파트에서 정의된 용어 목록을 추출 (`**용어(English)**` 패턴으로 검색).

### Phase 2: 20가지 패턴 검사

각 패턴의 검사 항목을 순서대로 적용하고, 발견된 문제를 기록합니다.

기록 형식:
```
| 패턴 | 위치(line) | 문제 내용 | 수정 방향 |
|------|-----------|----------|----------|
| E1   | L16       | recap→정의, 동기 없음 | 동기 문장 삽입 |
| F1   | L18       | "드로우콜" Part 1에서 이미 정의 | 재정의 형식 제거 |
| G4   | L777      | front-to-back → LRZ 오귀속 | Early-Z로 교정 |
```

**이 단계에서는 Edit를 실행하지 않습니다.** 전체 검사를 먼저 완료합니다.

### Phase 3: 수정 실행

Phase 2의 기록을 바탕으로 `Edit`를 적용합니다.

**수정 순서:**
1. **시리즈 일관성** (F1~F3) — 파트 간 모순/재정의/범위 침범을 먼저 해결.
2. **범위·정확성** (G1~G6) — 귀속/한정/인과/역할 오류를 수정.
3. **구조적 연결** (E1~E5) — 동기 부재, 단문 나열, 문단 분리, 전환, 밀도를 해결.
4. **문장 수준** (H1~H6) — 인과, 과적재, 중복, 순서, 단어, 전제를 수정.

**수정 시 절대 규칙:**
- **정보 손실 금지** — 삭제·축약 시 해당 정보가 글의 다른 위치에 존재하는지 반드시 확인.
- **기술 용어 보존** — 수정 전후를 대조하여 기술 용어가 탈락하지 않았는지 확인.
- **구체적 수치 보존** — 특정 숫자, 메커니즘, 단계별 설명이 사라지지 않았는지 확인.

### Phase 4: 자기 검증

1. 수정된 텍스트를 `Read`로 다시 읽는다.
2. 20가지 패턴을 재검사한다. 위반이 남아 있으면 추가 `Edit`로 교정한다.
3. **정보 손실 점검** — Phase 2에서 기록한 각 수정 항목에 대해, 해당 정보가 글의 어딘가에 존재하는지 확인한다.
4. **시리즈 일관성 재확인** — 수정으로 인해 이전/이후 파트와 새로운 모순이 생기지 않았는지 확인한다.
