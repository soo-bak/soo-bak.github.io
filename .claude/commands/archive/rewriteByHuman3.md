# 구조·맥락·정확성·자연스러움 통합 점검 (Rewrite by Human 3)

사람이 직접 읽었을 때 느끼는 **부자연스러움**을 탐지하고 수정합니다. 기술 정확성 훼손, 인과 오류, 구조 결함, 흐름 단절, 의미 반복, 문체 경직 — 기계적 검사로는 잡히지 않지만 독자가 "뭔가 이상하다"고 느끼는 문제를 다룹니다.

**독립 스킬** — 파이프라인에 속하지 않으며, 다른 스킬을 참조하거나 의존하지 않습니다. 단독으로 실행합니다.

## 인자

- `$ARGUMENTS`: 수정할 파일 경로 (선택사항). 없으면 사용자에게 파일 또는 텍스트를 요청합니다.

> **교차 규칙**: `/soobak-writing-review`의 **사용자 문체 규칙**이 이 스킬의 모든 수정에 적용됩니다. 충돌 시 문체 규칙이 우선합니다.

---

## 실행 원칙

**이 스킬이 호출되면 항상 4단계 전체를 빠짐없이 실행한다.** 글이 이미 깔끔해 보이거나 변경할 것이 적어 보여도, 프로세스를 축약하거나 실행이 불필요하다고 판단하지 않는다.

---

## 검사 패턴 (6카테고리, 27패턴)

모든 패턴은 하나의 원칙에서 비롯됩니다:

> **사람이 읽었을 때, 정확하고, 논리적이고, 흐름이 자연스러워야 한다.**

---

### A. 기술 정확성 훼손

수정·간결화·문체 개선 과정에서 원문의 정확성이 손상되는 패턴. 다른 모든 카테고리보다 우선하여 점검한다.

#### A1. 구체적 기술 용어 탈락

문체를 다듬거나 간결화하면서 구체적 기술 용어가 상위 개념으로 대체되는 패턴.

**검사 항목:**
- 수정 전에 있던 구체적 기술 용어(프래그먼트 셰이더, 상수 버퍼, 드로우콜 등)가 수정 후 상위 개념(셰이더, 버퍼, 호출 등)으로 바뀌었는가?
- 구체성이 필요한 맥락에서 의도적으로 추상화한 것인가, 실수로 탈락한 것인가?

**원칙:** 기술 글에서 구체적 용어는 정확성의 핵심. 문체를 다듬더라도 용어의 구체성은 보존한다.

**사례:**
- Before: "**프래그먼트 셰이더**가 여러 조명을 순회"
- After (잘못): "**셰이더**가 여러 조명을 순회" (← 버텍스 셰이더도 포함되는 모호한 표현)
- After (올바름): "**프래그먼트 셰이더**가 여러 조명을 순회" (원래 용어 유지)

#### A2. 핵심 정의·설명 탈락

간결화 과정에서 개념의 핵심 정의나 동작 설명이 누락되는 패턴.

**검사 항목:**
- 수정 전에 있던 정의("반복 횟수가 실행 중에 결정되는")가 수정 후 사라졌는가?
- 해당 정의가 독자의 이해에 필수적인가?

**원칙:** 간결함을 위해 정보를 줄일 때, 독자가 개념을 이해하는 데 필요한 핵심 정의는 반드시 유지한다.

**사례:**
- Before: "이 루프는 반복 횟수가 실행 중에 결정되는 **동적 루프**여야 합니다"
- After (잘못): "이 루프는 **동적 루프**여야 합니다" (← 동적 루프가 무엇인지 알 수 없음)

#### A3. 존재하지 않는 용어 생성

설명의 편의를 위해 실제 기술 체계에 없는 용어를 만들어 사용하는 패턴.

**검사 항목:**
- 볼드나 강조로 도입된 용어가 실제 공식 문서, API, 기술 문헌에 존재하는가?
- 설명을 위해 즉석에서 만든 용어인가?

**원칙:** 기존 기술 용어를 사용하거나, 용어 없이 서술한다. 없는 용어를 만들지 않는다.

**사례:**
- Before (잘못): "**글로벌 조명 버퍼**에 기록합니다" (← 이런 용어는 없음)
- After: "GPU 메모리에 기록합니다"

#### A4. 메커니즘 부정확 서술

데이터 흐름, 동작 순서, 인과 관계 등 기술적 메커니즘을 잘못 서술하는 패턴.

**검사 항목:**
- 서술된 데이터 흐름(무엇이 어디에서 어디로)이 실제와 일치하는가?
- 동작 순서(A 후 B)가 실제와 일치하는가?
- "오브젝트별로 조명 데이터를 기록" 같이 단계가 뒤섞이거나 축약되어 부정확해진 곳이 있는가?

**원칙:** 메커니즘은 각 단계를 정확히 서술한다. 축약이 부정확을 만들면 풀어서 쓴다.

**사례:**
- Before (잘못): "CPU가 오브젝트별로 선별한 조명 데이터를 상수 버퍼에 기록" (← 데이터가 아니라 인덱스+개수만 기록)
- After: "CPU는 프레임마다 모든 추가 조명의 데이터를 GPU 메모리에 기록합니다. 이후 드로우콜마다 선별된 조명의 인덱스와 개수를 상수 버퍼에 설정하면..."

#### A5. 일반 원리와 특정 구현 혼동

일반적 렌더링 원리를 특정 엔진의 구현인 것처럼, 또는 특정 엔진의 구현을 일반 원리인 것처럼 서술하는 패턴.

**검사 항목:**
- 일반 원리를 설명하면서 특정 엔진만의 구현처럼 서술하는가?
- 반대로, 특정 엔진의 고유 구현을 일반 원리처럼 단정하는가?

**원칙:** 일반 원리와 특정 구현을 구분한다. 특정 구현은 주체를 명시한다.

**사례:**
- Before (잘못): "씬의 모든 조명이 ForwardAdd 패스를 생성하지는 않습니다." (← Unity Built-in의 구현을 일반 사실처럼 단정)
- After: "이 비용을 줄이기 위해 **Unity의 Built-in 파이프라인은** 모든 조명에 ForwardAdd 패스를 생성하는 대신, 조명을 품질 등급으로 나누어 처리합니다."

### B. 인과/논리 오류

#### B1. 효과를 이유로 제시

"~하므로 X가 Y보다 앞에 위치한다" 형태에서, 제시된 이유가 실제로는 결과/효과인 패턴.

**검사 항목:**
- "~하므로", "그래서", "때문에" 뒤의 근거가 실제로는 결과/부수 효과인가?

**원칙:** 이유는 구조적 의존성(입출력 관계, 선행 조건)이어야 한다. 효과는 이유가 아니다.

**사례:**
- Before: "작업량이 감소합니다. **그래서** 컬링이 배칭보다 앞에 위치합니다."
- After: "컬링이 **무엇을** 렌더링할지 결정하면, 배칭은 그 결과를 받아 드로우콜을 묶습니다."

#### B2. 대상·역할 혼동

비용/문제 자체를 수단으로, 또는 수단을 비용으로 서술하는 패턴.

**검사 항목:**
- 개념의 역할(원인/결과, 수단/대상, 입력/출력)이 문장 내에서 뒤바뀌어 있는가?

**원칙:** 비용/문제 자체와, 그것을 줄이는 수단/기법을 구분한다.

#### B3. 동일 개념의 모순 서술

같은 개념을 서로 다른 문장에서 상충하게 서술하는 패턴.

**검사 항목:**
- 같은 개념의 위치/역할/특성이 글의 다른 부분에서 모순되는가?

**원칙:** 하나의 개념은 글 전체에서 일관된 위치/역할을 가진다.

#### B4. 부적절한 단어 선택

맥락에 맞지 않는 단어를 사용하여 의미가 왜곡되는 패턴.

**검사 항목:**
- "해법"(문제를 푸는 것) vs "방식"(방법), "사용합니다"(현재) vs "택했습니다"(과거) 등 맥락과 뉘앙스가 맞지 않는 단어가 있는가?
- 시제가 서술하는 시점과 일치하는가?

**원칙:** 단어의 뉘앙스와 시제가 맥락에 정확히 부합해야 한다.

**사례:**
- Before: "멀티패스는 당시에는 가장 현실적인 **해법**이었습니다" (← 멀티패스는 문제를 해결한 게 아니라 제약 안에서 택한 방법)
- After: "멀티패스 **방식을 택할 수밖에 없었습니다**"

### C. 구조 문제

#### C1. 소제목과 내용 범위 불일치

소제목이 특정 범위를 암시하지만 내용은 더 넓거나 좁은 범위를 다루는 패턴.

**검사 항목:**
- 소제목에 특정 대상이 명시되어 있는데 내용이 그 범위를 넘어서는가?
- 소제목이 포괄적인데 내용이 특정 대상만 다루는가?

**원칙:** 소제목이 약속한 범위와 내용이 다루는 범위가 일치해야 한다.

**사례:**
- Before: "## Built-in 렌더 파이프라인 -- Forward Rendering" → 내용은 URP까지 다룸
- After: "## 포워드 렌더링 (Forward Rendering)"

#### C2. 섹션 도입부와 하위 내용 중복

섹션 도입부의 일반적 서술을 하위 소제목/문단이 같은 추상 수준에서 반복하는 패턴.

**검사 항목:**
- 섹션 도입부(##)와 첫 번째 소제목(###) 아래가 같은 내용을 다른 말로 반복하는가?
- 도입부와 하위가 같은 추상 수준인가? (한쪽이 더 구체적이면 역할 분담이므로 문제 아님)

**원칙:** 도입부가 구체적이면 하위에서 반복하지 않는다. 같은 추상 수준의 반복은 병합한다.

**사례:**
- Before: "## URP의 렌더링 구조 — 단계들을 렌더 패스로 구성하여 실행" + "### Renderer와 Render Pass — 각 렌더 패스는 하나의 목적만 담당"
- After: 도입부에서 구체적 단계를 풀어 설명 → 하위에서는 Renderer의 역할만 다룸

#### C3. 미설명 용어의 선행 사용

정의/설명 전에 전문 용어를 사용하는 패턴.

**검사 항목:**
- 볼드나 괄호 영문 없이 전문 용어가 등장하고, 정의가 뒤에 나오는가?
- 독자가 해당 용어를 처음 만나는 시점에서 의미를 알 수 있는가?

**원칙:** 용어는 정의 후에 사용한다.

**사례:**
- Before: (예시에서) "**픽셀 라이트** 3개" → (아래에서 비로소) "픽셀 라이트란..."
- After: (예시에서) "조명 3개" → (아래에서 처음으로) "**픽셀 라이트(Pixel Light)**"

#### C4. 차이점 예고가 유일성 암시

"차이는 ~에 있습니다" 형태가 해당 항목이 유일한 차이인 것처럼 읽히는 패턴.

**검사 항목:**
- "두 X의 차이는 ~입니다" 형태에서, 실제로는 다른 차이점도 있는데 유일한 것처럼 읽히는가?

**원칙:** 차이점 예고가 유일성을 암시하면 삭제하거나 "여러 차이 중 ~가 가장 직접적이다" 등으로 한정한다. 아래 소제목에서 차이가 자연스럽게 드러나면 예고 자체가 불필요할 수 있다.

**사례:**
- Before: "두 파이프라인의 차이는 조명을 몇 번의 패스로 처리하는지에 있습니다."
- After: (삭제 — 아래 ### 멀티패스 포워드 렌더링에서 자연스럽게 차이가 드러남)

#### C5. 맥락 전환에 구조적 분리 필요

이전 주제와 다른 새 주제가 같은 섹션 안에 소제목 없이 끼어 있는 패턴.

**검사 항목:**
- 한 소제목 안에서 주제가 전환되는데, 별도의 소제목으로 분리되지 않은 곳이 있는가?
- 독자가 "갑자기 다른 이야기가 나왔다"고 느끼는가?

**원칙:** 주제가 전환되면 소제목으로 분리한다.

**사례:**
- Before: "### 멀티패스의 비용 계산" 안에서 비용 공식 → 표 → 갑자기 "실제로는 모든 조명이 ForwardAdd를 생성하지는 않습니다" (조명 품질 등급이라는 새 주제)
- After: 표 뒤에 "### 조명의 품질 등급"으로 분리

### D. 흐름 단절

#### D1. 불필요한 되짚기/회귀

"앞서", "앞에서 살펴본 것처럼" 등으로 같은 글의 이전 섹션을 되짚는 패턴.

**검사 항목:**
- "앞서", "위에서", "앞에서 ~라고 했습니다" 등의 회귀 표현이 있는가?
- 회귀 없이 직접 서술하거나 대조할 수 있는가?
- **예외:** 다른 글(다른 URL)로의 참조는 독자가 해당 글을 읽지 않았을 수 있으므로 허용.

**원칙:** 같은 글 안에서의 회귀는 삭제한다. 독자는 바로 위를 방금 읽었다.

**사례:**
- Before: "**앞에서 살펴본 것처럼** URP의 렌더 패스는 정해진 순서대로 실행됩니다."
- After: "URP의 렌더 패스는 정해진 순서대로 실행됩니다."

#### D2. 확립된 정보의 불필요한 반복

이미 확립된 사실을 다른 표현으로 다시 서술하는 패턴.

**검사 항목:**
- 이전 섹션/문단에서 이미 설명·확립된 사실을 다시 서술하는가?
- 새로운 정보가 추가되는가, 단순 반복인가?

**원칙:** 확립된 사실은 반복하지 않는다. 새 섹션은 새 정보로 시작한다.

**사례:**
- Before: (이미 "조명 수에 비례하여 드로우콜 증가" 확립 후) "멀티패스 포워드 렌더링에서는 픽셀 라이트가 많을수록 드로우콜이 늘어나므로" (← 같은 말의 반복)
- After: (삭제)
- Before: (이미 ForwardBase가 메인 라이트 처리한다고 설명 후) "가장 밝은 픽셀 라이트 하나는 ForwardBase 패스에서 메인 라이트로 처리되고" (← 중복)
- After: (삭제)

#### D3. 문제 인식 없는 해결책 전환

문제를 인식시키지 않고 바로 해결책/대응 방식을 제시하는 패턴.

**검사 항목:**
- "이 비용을 줄이기 위해", "이 문제를 해결하기 위해" 직전에, 독자가 해당 비용/문제를 실감할 수 있는 서술이 있는가?
- 구체적 수치나 예시만 있고 "이것이 문제다"라는 인식 없이 바로 해결책으로 넘어가는가?

**원칙:** 해결책 전에 문제가 먼저 독자에게 와닿아야 한다.

**사례:**
- Before: (표로 30회 드로우콜 계산) → (바로) "### 조명의 품질 등급 — 이 비용을 줄이기 위해..."
- After: (표 뒤에) "조명이 늘어날수록 드로우콜이 급격히 증가하므로, 성능 병목이 됩니다." → "### 조명의 품질 등급"

#### D4. 순환 구조

A를 B로 설명하고, B를 다시 A로 설명하여 독자가 맴도는 패턴.

**검사 항목:**
- 연결 문장에서 개념 A를 B로 정의하고, 바로 아래에서 B를 다시 A로 정의하는가?
- 두 문장을 읽은 뒤에도 "결국 뭔데?"라는 질문이 남는가?

**원칙:** 설명은 직선이어야 한다. A → B → C로 진행하되, A → B → A로 돌아오지 않는다.

**사례:**
- Before: (연결) "조명 수는 ForwardAdd를 생성하는 조명만 해당" + (섹션) "픽셀 라이트만 ForwardAdd를 생성" (← 같은 말을 방향만 바꿔 반복)
- After: (연결 제거, 섹션이 직접 설명) "이 비용을 줄이기 위해 Unity의 Built-in 파이프라인은 조명을 품질 등급으로 나누어 처리합니다. ... 이 중 ForwardAdd 패스를 생성하는 것은 픽셀 라이트뿐입니다."

#### D5. 중간 정보가 흐름을 끊음

A → B로 이어져야 할 흐름 사이에 부차적 정보 C가 끼어드는 패턴.

**검사 항목:**
- A 설명 후 B 설명으로 자연스럽게 이어져야 하는데, 중간에 예외 사항이나 부차적 정보가 끼어 있는가?
- 해당 중간 정보를 뒤로 옮겨도 논리가 성립하는가?

**원칙:** 핵심 흐름을 먼저 완성하고, 부차적 정보는 뒤에 배치한다.

**사례:**
- Before: ForwardBase 설명 → "조명이 하나뿐이면 이 패스만으로 완료" → ForwardAdd 설명 (← 예외 사항이 메인 흐름을 끊음)
- After: ForwardBase 설명 → ForwardAdd 설명 → 가산 블렌딩 → "조명이 하나뿐이면 ForwardAdd가 실행되지 않으므로, ForwardBase만으로 완료"

#### D6. 관련 설명의 불필요한 분리

하나의 흐름이어야 할 내용이 별도 문장이나 문단으로 끊겨 있는 패턴.

**검사 항목:**
- 하나의 동작/과정을 설명하는 관련 문장들이 빈 줄이나 별도 문단으로 분리되어 있는가?
- 합쳐서 한 문장/문단으로 만들면 더 자연스러운가?

**원칙:** 하나의 동작/과정은 하나의 흐름으로 연결한다.

**사례:**
- Before: "ForwardAdd는 같은 오브젝트를 다시 그립니다." (빈 줄) "ForwardBase의 픽셀 색상 위에 가산 블렌딩으로 누적됩니다."
- After: "ForwardAdd는 같은 오브젝트를 다시 그리면서 해당 조명만 계산하고, 그 결과를 ForwardBase의 픽셀 색상 위에 가산 블렌딩으로 누적합니다."

#### D7. 주어 없는 갑작스러운 전환

주어나 맥락 없이 갑자기 새로운 서술이 시작되는 패턴.

**검사 항목:**
- 문장의 주어가 생략되어, 무엇에 대한 이야기인지 파악하려면 앞 문장을 다시 읽어야 하는가?
- "같은 X이지만", "다만" 등으로 시작하는데, 무엇이 "같은" 것인지 명시되지 않는가?

**원칙:** 문맥 전환이 있을 때는 주어를 명시한다.

**사례:**
- Before: "같은 포워드 렌더링이지만, 조명을 몇 번의 패스로 처리하는지가 다릅니다." (← 무엇이 "같은" 것인지 모호)
- After: "다만 **Built-in과 URP는** 조명을 몇 번의 드로우콜로 처리하는지가 다릅니다."

#### D8. 불필요한 범위/메타 선언

맥락에서 이미 암시된 글의 범위나 구조를 명시적으로 선언하는 패턴.

**검사 항목:**
- "이 시리즈에서는 ~를 다룹니다", "이 글에서는 ~를 살펴봅니다" 등의 범위 선언이, 앞뒤 맥락에서 이미 충분히 드러나는가?
- 해당 문장을 삭제해도 독자가 글의 범위를 파악하는 데 지장이 없는가?

**원칙:** 맥락에서 이미 드러나는 범위를 별도로 선언하지 않는다.

**사례:**
- Before: (URP/HDRP/Built-in 소개 직후) "이 시리즈에서는 모바일 최적화에 초점을 맞추어 Built-in과 URP를 중심으로 다룹니다." (← HDRP가 "고사양 PC/콘솔"이라고 이미 구분됨)
- After: (삭제 — 다음 문장 "Built-in과 URP는..."에서 자연스럽게 범위가 좁혀짐)

### E. 동어 반복

#### E1. 의미 반복

한 문단 안에서 같은 의미가 형태만 바뀌어 반복되는 패턴.

**검사 항목:**
- **(원인-결과 동어)** 원인과 결과가 같은 말인가?
- **(표현 교체 반복)** 같은 의미를 다른 표현으로 두 번 이상 쓰는가?
- **(다단계 반복)** 같은 축소/증가 의미를 3번 이상 반복하는가?
- **(용어 뜻 풀이 중복)** 용어 자체에 이미 담긴 뜻을 다시 풀어쓰는가?

**원칙:** 가장 명확한 표현 하나를 남기고 나머지를 삭제한다.

**사례 (용어 뜻 풀이):**
- Before: "색상값을 더하는 **가산 블렌딩(Additive Blending)**" (← "가산"이 이미 "더하는" 뜻)
- After: "**가산 블렌딩(Additive Blending)**"

#### E2. 섹션 경계 동어 반복

새 섹션의 첫 문장이 이전 섹션의 결론/내용을 요약 반복하는 패턴.

**검사 항목:**
- 새 섹션의 첫 1~2문장이 이전 섹션에서 이미 설명한 내용을 되풀이하는가?

**원칙:** 새 섹션은 새 정보로 시작한다. 독자는 바로 위 섹션을 방금 읽었다.

#### E3. 소제목·도입문 중복

소제목이 이미 전달한 신호를 도입문이 반복하는 패턴.

**검사 항목:**
- 소제목에 "제약", "장점" 등이 있는데, 도입문에서 같은 말을 하는가?

**원칙:** 소제목 자체가 전환 신호. 도입문에서 같은 말을 반복하지 않는다.

#### E4. 정의 분산

같은 개념의 정의가 인접 문장 여러 곳에 나뉘어 있는 패턴.

**검사 항목:**
- 같은 용어/개념이 인접한 2개 이상의 문장에서 각각 부분적으로 정의되는가?

**원칙:** 하나의 개념은 한 곳에서 완결적으로 정의한다.

#### E5. 같은 예시의 반복 사용

동일한 구체적 예시가 글에서 2회 이상 같은 목적으로 사용되는 패턴.

**원칙:** 예시는 한 번만. 두 번째 등장이 같은 원리의 반복이면 참조로 대체한다.

#### E6. 시각 요소 직후 산문 재나열

표·다이어그램 직후의 산문이 시각 요소의 내용을 그대로 반복 나열하는 패턴.

**원칙:** 산문은 시각 요소에 없는 새로운 맥락·해석만 추가한다.

**주의:** 개념 다이어그램 내부의 "→" 결론 라인은 중복이 아닙니다. 다이어그램을 자체 완결적으로 만드는 요소이므로 유지합니다.

#### E7. 중복 볼드 소개

같은 용어를 여러 곳에서 **볼드 + 괄호 영문**으로 재소개하는 패턴.

**검사 항목:**
- 같은 용어가 글에서 2회 이상 **볼드 + 괄호 영문**으로 소개되는가?

**원칙:** 용어의 볼드 소개는 최초 1회만. 이후에는 볼드 없이 사용한다.

**사례:**
- Before: (L181) "**픽셀 라이트(Pixel Light)**" + (L191) "**정점 라이트(Vertex Light)** 또는 **SH 라이트(Spherical Harmonics Light)**로 강등"
- After: (L191) "정점 라이트 또는 SH 라이트로 강등"

### F. 문체 경직

#### F1. 교과서식 나열 도입문

"~에는 A, B, C가 있습니다" 형태의 목록 예고 패턴.

**원칙:** 목록 예고 없이 바로 항목 설명으로 진입한다.

#### F2. 불필요한 나열/펼침

상위 개념으로 요약 가능한 내용을 하나하나 펼쳐 쓰는 패턴.

**원칙:** 개별 항목의 구분이 중요하지 않으면 상위 개념으로 요약한다.

#### F3. 이질적 항목의 모호한 병합

역할이 다른 항목들을 "이와 함께", "그 외에도" 등으로 이어붙이는 패턴.

**원칙:** 역할이 다르면 역할 기준으로 구분한다.

#### F4. 추상적 나열 (구체성 부족)

독자가 각 항목의 의미를 직관적으로 파악할 수 없는 추상적 나열 패턴.

**검사 항목:**
- "A, B, C 등 여러 단계가 필요합니다" 형태에서, A/B/C가 단어만 나열되어 있고 각각이 무엇을 하는지 설명이 없는가?
- 도입부가 구체적 설명을 담당해야 하는 위치인데 추상적 나열로만 되어 있는가?

**원칙:** 도입부가 구체적 설명을 담당해야 할 위치라면, 추상적 나열 대신 각 항목의 역할/동작을 풀어서 설명한다.

**사례:**
- Before: "프레임을 완성하려면 조명 외에도 깊이 기록, 투명 처리, 후처리 등 여러 단계가 필요합니다."
- After: "프레임을 완성하려면 조명 외에도 여러 단계가 필요합니다. 깊이값을 기록하는 단계, 불투명 오브젝트를 그리는 단계, 투명 오브젝트를 그리는 단계, 후처리를 적용하는 단계 등이 각각 별도로 실행됩니다."

#### F5. 끊어진 문장들의 나열

하나의 문단이어야 할 관련 문장들이 빈 줄로 분리되어 나열처럼 읽히는 패턴.

**검사 항목:**
- 빈 줄로 분리된 2~3개의 짧은 문장이, 실제로는 하나의 논점을 이루는가?
- 하나의 문단으로 합치면 더 자연스러운가?

**원칙:** 하나의 논점을 구성하는 관련 문장들은 하나의 문단으로 연결한다.

**사례:**
- Before: "이 시리즈에서는 모바일 최적화에 초점을 맞추어 Built-in과 URP를 중심으로 다룹니다." (빈 줄) "두 파이프라인은 셰이더 시스템..." (빈 줄) "이 글에서는 조명 처리 방식의 차이를..."
- After: "Built-in과 URP는 셰이더 시스템, 포스트 프로세싱, 카메라 구조 등 여러 면에서 다르지만, 렌더링 성능에 가장 직접적인 영향을 미치는 차이는 **조명 처리 방식**과 **배칭 구조**입니다. 이 글에서는 조명 처리 방식의 차이를 다루고, 배칭은 Part 2에서 이어집니다."

#### F6. 일관성 부족 (용어·표기)

같은 대상을 다른 표기로 지칭하거나, 기호/음차 사용이 일관되지 않는 패턴.

**검사 항목:**
- 같은 기능/대상을 다른 이름으로 지칭하는 곳이 있는가? (예: "모바일 우선 설계" vs "싱글패스, SRP Batcher, 경량 설계")
- 널리 알려진 기술 용어의 한글 음차와 영어 표기가 글 안에서 혼용되는가? (예: "볼류메트릭 라이팅" vs "Volumetric Lighting")
- 곱셈 기호("x" vs "×") 등 기호 사용이 일관되는가?

**원칙:** 같은 대상은 같은 표기로, 기호와 용어는 글 전체에서 일관되게.

## 작업 프로세스

### Phase 1: 글 전체 파악

1. `$ARGUMENTS`가 있으면 해당 파일을 `Read`로 전체 읽기. 없으면 사용자에게 파일 경로 요청.
2. `Grep`으로 `##`/`###` 제목을 추출하여 **섹션 지도** 작성.
3. 각 섹션의 역할 식별: 도입, 개념 소개, 상세 설명, 예시, 비교, 마무리.
4. 시각 요소(표, SVG/ASCII 다이어그램, 코드 블록) 위치와 내용을 목록화.

### Phase 2: 27가지 패턴 검사

각 패턴의 검사 항목을 순서대로 적용하고, 발견된 문제를 기록합니다.

기록 형식:
```
| 패턴 | 위치(line) | 문제 내용 | 수정 방향 |
|------|-----------|----------|----------|
| A1   | L97       | "프래그먼트 셰이더" → "셰이더"로 탈락 | 원래 용어 복원 |
| C1   | L93       | "Built-in -- Forward" 소제목에 URP 내용 포함 | 소제목 범위 확장 |
| D4   | L178      | 연결문과 본문이 순환 설명 | 연결문 제거 |
```

**이 단계에서는 Edit를 실행하지 않습니다.** 전체 검사를 먼저 완료합니다.

### Phase 3: 수정 실행

Phase 2의 기록을 바탕으로 `Edit`를 적용합니다.

**수정 순서:**
1. **기술 정확성** (A1~A5) — 용어·정의·메커니즘의 정확성을 먼저 확보. 정확하지 않으면 다른 수정이 무의미.
2. **인과/논리** (B1~B4) — 인과 오류, 역할 혼동, 모순, 단어 선택을 바로잡는다.
3. **구조** (C1~C5) — 소제목·범위, 도입부 중복, 미설명 용어, 차이점 예고, 맥락 전환을 정리. 구조가 맞아야 흐름을 판단할 수 있다.
4. **흐름** (D1~D8) — 되짚기, 반복, 순환, 흐름 끊김, 주어 누락 등을 수정.
5. **동어 반복** (E1~E7) — 의미 반복, 정의 분산, 중복 볼드 등을 정리.
6. **문체** (F1~F6) — 교과서식 나열, 추상적 나열, 끊어진 문장, 일관성을 수정.

**수정 시 절대 규칙:**
- **정보 손실 금지** — 삭제·축약 시 해당 정보가 글의 다른 위치(또는 시각 요소)에 존재하는지 반드시 확인. 글 전체에서 사라지는 정보가 있으면 안 됨.
- **기술 용어 보존** — 수정 과정에서 기술 용어가 탈락하지 않았는지 before/after를 대조.
- **구체적 수치 보존** — 특정 숫자, 메커니즘, 단계별 설명이 수정 과정에서 사라지지 않았는지 확인.
- **수정이 새로운 패턴 위반을 만들지 않는지 확인** — 특히 A1(용어 탈락), A2(정의 탈락)은 수정 과정에서 가장 빈번하게 발생.

### Phase 4: 자기 검증 + 보고

1. 수정된 텍스트를 `Read`로 다시 읽는다.
2. 27가지 패턴을 재검사한다. 위반이 남아 있으면 추가 `Edit`로 교정한다.
3. **정보 손실 점검** — Phase 2에서 기록한 각 수정 항목에 대해, 해당 정보가 글의 어딘가에 존재하는지 확인한다.
4. 결과를 아래 형식으로 보고한다.
