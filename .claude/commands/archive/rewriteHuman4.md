# 탑다운 스케일 검증 (Rewrite by Human 4)

수정된 결과물 자체가 **모든 스케일에서 자연스러운지** 검증합니다. 안티패턴 탐지가 아니라, 텍스트가 적절하고 자연스럽고 정확하고 직관적이며 간결하면서도 완전한지 확인합니다.

**독립 스킬** — 파이프라인에 속하지 않으며, 다른 스킬(v1~v3 포함)을 참조하거나 의존하지 않습니다. 단독으로 실행합니다.

## 인자

- `$ARGUMENTS`: 수정할 파일 경로 (선택사항). 없으면 사용자에게 파일 또는 텍스트를 요청합니다.

> **교차 규칙**: `/soobak-writing-review`의 **사용자 문체 규칙**이 이 스킬의 모든 수정에 적용됩니다. 충돌 시 문체 규칙이 우선합니다.

---

## 실행 원칙

**이 스킬이 호출되면 항상 4단계 전체를 빠짐없이 실행한다.** 글이 이미 깔끔해 보이거나 변경할 것이 적어 보여도, 프로세스를 축약하거나 실행이 불필요하다고 판단하지 않는다.

---

## 5단계 스케일 검사 (탑다운)

모든 스케일에서 동일한 5가지 기준을 적용합니다:

> **적절성 · 자연스러움 · 정확성 · 직관적 전달 · 간결+완전**

큰 스케일부터 확인해야 작은 수정이 유효합니다. 스케일 1부터 5까지 순서대로 검사합니다.

---

### 스케일 1. 글 단위

글 전체를 하나의 덩어리로 보고 검사합니다.

**검사 항목:**

1. **논리 구조 일관성** — 글 전체의 논리 구조가 일관적인가. 도입부에서 설정한 전제가 본론과 마무리까지 유지되는가.
2. **도입→본론→마무리 흐름** — 도입이 본론으로 자연스럽게 이어지는가. 마무리가 본론의 내용을 반영하는가. 도입에서 약속한 것을 본론에서 다루는가.
3. **개념·예시 일관성** — 같은 개념이 글 전체에서 동일한 의미·표기로 사용되는가. 예시가 일관된 시나리오를 따르는가.
4. **톤·밀도 균일성** — 수정 전후로 글 전체의 톤과 설명 밀도가 균일한가. 특정 섹션만 지나치게 조밀하거나 성긴 곳이 없는가.

---

### 스케일 2. 흐름과 맥락 단위

섹션 간, 문단 간의 연결을 검사합니다.

**검사 항목:**

1. **섹션 간 이음새** — 각 섹션의 끝과 다음 섹션의 시작이 자연스럽게 이어지는가. 갑작스러운 주제 전환이 없는가.
2. **삭제·추가의 부작용** — 삭제/추가된 문장이 주변에 빈자리·이질감을 만들지 않는가. 삭제 후 앞뒤가 자연스럽게 붙는가.
   - **사례 (삭제가 새 문제 생성):**
     - Before: 도입문 "카메라의 시야는 무한하지 않습니다." 삭제 → 소제목 직후 갑자기 기술 설명 시작
     - 문제: 도입문이 없어지면서 독자가 맥락 없이 본론에 던져짐
     - 원칙: 문장을 삭제할 때는 삭제 후의 흐름을 반드시 확인. 빈자리가 생기면 대체 문장을 넣거나, 다음 문장을 수정하여 자연스럽게 연결.
3. **문단 전환의 자연스러움** — 문단이 바뀔 때 독자가 "왜 갑자기?"라고 느끼지 않는가. 전환이 논리적으로 필연적인가.
4. **정보 제시 순서** — 독자의 사고 흐름과 일치하는가. 결론이 근거보다 먼저 나오거나, 정의가 사용 후에 등장하지 않는가.

---

### 스케일 3. 소제목 단위

소제목과 그 아래 내용의 관계를 검사합니다.

**검사 항목:**

1. **소제목과 첫 문장의 연결** — 소제목이 설정하는 주제로 첫 문장이 자연스럽게 진입하는가.
   - **사례 (연결 실패):**
     - Before: "## Frustum Culling" → "카메라의 시야는 각도와 거리 양쪽에서 제한됩니다."
     - 문제: 소제목은 "Frustum Culling"이라는 기법을 예고하지만, 첫 문장은 "카메라의 시야"라는 다른 주어로 시작. 독자가 소제목과 첫 문장 사이에서 연결고리를 찾아야 함.
     - After: "## Frustum Culling" → "**Frustum Culling**은 카메라의 시야 영역, 즉 **프러스텀(Frustum)** 바깥에 있는 오브젝트를 렌더링 대상에서 제외하는 기법입니다."
     - 원칙: 소제목의 핵심 키워드가 첫 문장에 등장하거나, 첫 문장이 소제목의 주제를 직접 받아야 한다.
2. **소제목의 기대 충족** — 소제목이 설정하는 기대를 내용이 충족하는가. "## 장점"인데 단점도 섞여 있지 않은가.
3. **소제목 위계와 내용 위계** — `##`과 `###`의 수준이 내용의 포함 관계와 일치하는가. `###`이 `##`의 하위 주제인가.

---

### 스케일 4. 문장 단위

개별 문장의 구조와 톤을 검사합니다.

**검사 항목:**

1. **한국어 문장 구조** — 주어-목적어-서술어 순서가 한국어로 자연스러운가. 주어가 너무 길거나, 서술어가 주어에서 너무 멀지 않은가.
2. **병렬 구조의 자연스러움** — "~은 A, ~은 B" 형태의 병렬이 읽기 쉬운가.
   - **사례 (부자연스러운 병렬):**
     - Before: "**Near Clip Plane**은 카메라에서 가장 가까운, **Far Clip Plane**은 가장 먼 렌더링 경계면입니다."
     - 문제: 한국어에서 주어를 바꿔가며 서술어를 공유하는 구조는 부자연스러움. 특히 수식어("카메라에서 가장 가까운")가 길어지면 더 어색.
     - After: "**Near Clip Plane**은 카메라에서 가장 가까운 렌더링 경계면이고, **Far Clip Plane**은 가장 먼 렌더링 경계면입니다."
     - 원칙: 한국어 병렬은 각 절이 서술어를 갖추어야 자연스럽다. 서술어 공유를 시도하면 뒤의 절이 허공에 뜬다.
3. **도입문·전환문의 톤** — 너무 단정적이지도, 갑작스럽지도 않은가.
   - **사례 (과도하게 단정적인 톤):**
     - Before: "카메라의 시야는 무한하지 않습니다."
     - 문제: 부정문으로 시작하면 "당연한 걸 왜 말하지?"라는 느낌. 도입문의 역할은 독자를 주제로 이끄는 것인데, 단정적 부정은 벽을 세움.
     - After: "카메라가 한 번에 담을 수 있는 공간에는 한계가 있습니다."
     - 원칙: 도입문은 독자가 공감하거나 자연스럽게 따라올 수 있는 진술로 시작. 부정문("~않다", "~없다")보다 긍정적 진술이 도입에 적합한 경우가 많다.
4. **문장 길이** — 너무 긴 문장(3개 이상의 절)은 분리를 검토. 너무 짧은 문장이 연속되면 통합을 검토.
5. **수식어-피수식어 거리** — 수식어와 피수식어 사이에 다른 요소가 끼어 있지 않은가. 수식 관계가 즉시 파악되는가.

---

### 스케일 5. 단어 단위

개별 단어와 조사·어미의 적절성을 검사합니다.

**검사 항목:**

1. **단어 뉘앙스** — 단어의 뉘앙스가 맥락에 정확히 맞는가.
   - **사례 (뉘앙스 부적합):**
     - Before: "Near Clip Plane과 Far Clip Plane이 **이루는** 절두체"
     - 문제: "이루다"는 구성원이 모여 전체를 형성하는 뉘앙스("팀을 이루다"). 두 평면이 절두체를 "이루는" 것이 아니라, 여러 요소가 결합하여 절두체라는 공간을 "만들어내는/형성하는" 것.
     - After: "Near Clip Plane과 Far Clip Plane이 **만들어내는** 절두체"
     - 원칙: 단어의 뉘앙스를 맥락에서 검증. "이루다"(구성), "만들어내다"(생성/형성), "구성하다"(부분→전체) 등 유사 동사의 미묘한 차이를 구분.
2. **부정 표현 vs 긍정 표현** — 부정 표현("무한하지 않다", "불가능하지 않다")보다 긍정 표현이 자연스러운 곳은 없는가.
   - **사례:**
     - Before: "카메라의 시야는 무한하지 않습니다."
     - After: "카메라가 한 번에 담을 수 있는 공간에는 한계가 있습니다."
     - 원칙: 부정으로 우회하지 말고 긍정으로 직접 서술. 다만 대비("A와 달리 B는 ~않다")에서의 부정은 자연스러울 수 있으므로 맥락 판단.
3. **불필요한 단어 반복** — 같은 단어가 인접 문장에서 불필요하게 반복되지 않는가. 의도적 반복(강조)과 실수 반복을 구분.
4. **기술 용어 보존** — 수정 과정에서 기술 용어가 정확히 보존되었는가. 상위 개념으로 대체되지 않았는가.
5. **조사·어미 자연스러움** — "을/를", "이/가", "은/는" 선택이 맥락에 맞는가. 종결어미가 단조롭게 반복되지 않는가.

## 작업 프로세스

### Phase 1: 파악

1. `$ARGUMENTS`가 있으면 해당 파일을 `Read`로 전체 읽기. 없으면 사용자에게 파일 경로 요청.
2. `Grep`으로 `##`/`###` 제목을 추출하여 **섹션 지도** 작성.
3. 각 섹션의 역할 식별: 도입, 개념 소개, 상세 설명, 예시, 비교, 마무리.
4. 시각 요소(표, SVG/ASCII 다이어그램, 코드 블록) 위치와 내용을 목록화.

### Phase 2: 탑다운 검사

스케일 1→2→3→4→5 순서로 검사하고, 발견된 문제를 기록합니다.

기록 형식:
```
| 스케일 | 검사항목 | 위치(line) | 문제 내용 | 수정 방향 |
|--------|---------|-----------|----------|----------|
| 1. 글  | 톤 균일 | 전체      | 후반부가 전반부보다 설명이 성김 | 후반부 밀도 보강 |
| 3. 소제목 | 첫 문장 연결 | L71 | "## Frustum Culling" 직후 카메라 시야 이야기로 시작 | 소제목 키워드로 첫 문장 시작 |
| 4. 문장 | 병렬 구조 | L85 | "~은 가장 가까운, ~은 가장 먼" 서술어 공유 | 각 절에 서술어 부여 |
| 5. 단어 | 뉘앙스 | L90 | "이루는" → 구성 뉘앙스가 맞지 않음 | "만들어내는"으로 교체 |
```

**이 단계에서는 Edit를 실행하지 않습니다.** 전체 검사를 먼저 완료합니다.

### Phase 3: 수정

Phase 2의 기록을 바탕으로 `Edit`를 적용합니다.

**수정 순서 — 큰 스케일을 먼저 적용합니다.** 큰 수정이 작은 수정을 불필요하게 만들 수 있기 때문입니다.

1. **스케일 1 (글 단위)** — 전체 구조·톤·밀도 조정
2. **스케일 2 (흐름·맥락)** — 섹션 이음새, 삭제 부작용 수정, 전환 개선
3. **스케일 3 (소제목)** — 소제목↔첫 문장 연결, 기대 충족, 위계 조정
4. **스케일 4 (문장)** — 문장 구조, 병렬, 톤, 길이, 수식어 거리
5. **스케일 5 (단어)** — 뉘앙스, 부정→긍정, 반복, 용어 보존, 조사·어미

**수정 시 절대 규칙:**
- **정보 손실 금지** — 삭제·축약 시 해당 정보가 글의 다른 위치(또는 시각 요소)에 존재하는지 반드시 확인. 글 전체에서 사라지는 정보가 있으면 안 됨.
- **기술 용어 보존** — 수정 과정에서 기술 용어가 탈락하지 않았는지 before/after를 대조.
- **구체적 수치 보존** — 특정 숫자, 메커니즘, 단계별 설명이 수정 과정에서 사라지지 않았는지 확인.
- **수정이 새 문제를 만들지 않는지 확인** — 특히 스케일 2(삭제 부작용)와 스케일 3(소제목 연결)은 수정 과정에서 가장 빈번하게 발생.

### Phase 4: 자기 검증 + 보고

1. 수정된 텍스트를 `Read`로 다시 읽는다.
2. 5 스케일을 다시 검사한다. 위반이 남아 있으면 추가 `Edit`로 교정한다.
3. **정보 손실 점검** — Phase 2에서 기록한 각 수정 항목에 대해, 해당 정보가 글의 어딘가에 존재하는지 확인한다.
4. 결과를 아래 형식으로 보고한다.

## 자기 검증 체크리스트

모든 Edit 적용 후, 수정된 텍스트를 Read로 다시 읽고 아래 항목을 검증합니다:

1. **글 전체 일관성** — 논리 구조, 개념 사용, 톤이 글 전체에서 일관되는가?
2. **섹션 간 흐름** — 섹션의 끝과 다음 섹션의 시작이 자연스럽게 이어지는가?
3. **소제목↔첫 문장 연결** — 모든 소제목과 그 아래 첫 문장이 자연스럽게 연결되는가?
4. **문장 구조** — 부자연스러운 병렬, 과도한 문장 길이, 수식어 거리 문제가 없는가?
5. **단어 뉘앙스** — 맥락에 맞지 않는 단어 선택이 없는가?
6. **정보 손실** — 수정 전에 있던 기술 용어·수치·메커니즘이 글 전체에서 사라지지 않았는가?
7. **수정이 새 문제를 만들지 않음** — 수정으로 인해 앞뒤 문맥이 끊기거나, 빈자리가 생기거나, 참조가 깨진 곳이 없는가?
8. **전체 통독** — 수정된 글 전체를 처음부터 끝까지 하나의 글로 읽었을 때, 모든 스케일에서 자연스러운가?

{파일 경로}

### 스케일별 검사 결과
| 스케일 | 검사항목 | 탐지 건수 | 주요 사례 |

| 1. 글 단위 | 논리 구조 일관성 | N건 | ... |
