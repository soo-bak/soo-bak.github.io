# 글쓰기 수정 스킬 — 골격 (Structure)

주어진 글의 **뼈대**를 다듬습니다. 내용의 배치, 문장 구조, 용어 일관성, 간결함을 점검합니다.

## 파이프라인 위치

```
▶ /edit-writing → /clarify-writing → /refine-style → /smooth-flow → /verify-facts → /deep-review → /align-reference
    골격             이해               목소리          흐름            사실             독자 시뮬레이션    레퍼런스 정렬
```

이 스킬은 파이프라인의 **첫 단계**입니다. 글의 뼈대만 다루며, 다음은 다른 스킬이 담당합니다:
- 독자의 이해 → `/clarify-writing`
- 톤, 리듬, 서사 → `/refine-style`
- 읽기 흐름 → `/smooth-flow`
- 사실관계 → `/verify-facts`
- 독자 경험 통합 점검 → `/deep-review`
- 레퍼런스 정렬 → `/align-reference`

## 인자
- `$ARGUMENTS`: 수정할 파일 경로 (선택사항). 없으면 사용자에게 파일 또는 텍스트를 요청합니다.

> **교차 규칙**: `/soobak-writing-review`의 **사용자 문체 규칙**(질문형 금지, 기술 용어 보존, 메타 문장 금지, 교과서적 표현 금지, 산문 우선, 초보자 기준 설명, 추상적 서술 구체화)이 이 스킬의 모든 수정에 적용됩니다. 충돌 시 문체 규칙이 우선합니다.

### 대상 독자
**초보자**가 외부 자료 없이 이 글만으로 이해할 수 있는 수준을 기준으로 합니다. 문장 구조와 용어 사용 시 초보자의 이해를 저해하지 않아야 합니다.

## 수정 원칙

### 1. 용어 일관성
- 같은 개념에 같은 용어 일관 사용
- 새 용어는 첫 등장 시 정의
- 자연스러운 조합: 파장-길다/짧다, 주파수-높다/낮다

### 2. 간결함
- 불필요한 반복 제거
- 필요한 정보는 반드시 포함
- "없어도 의미 전달에 지장 없는 단어" 제거

### 3. 문장 명확성
- 짧은 문장 선호 (2줄 이내)
- 주어-서술어 관계 명확
- 대명사보다 명확한 지시어

### 4. 문단 구조
- 하나의 문단에 하나의 주제
- "첫째", "둘째"로 논리 구분
- 앞뒤 문단 연결 명시

### 5. 산문 우선
- 설명이 필요한 항목은 불릿 리스트가 아닌 산문(문단)으로 풀어 쓰기
- 불릿은 병렬 비교, 요약, 체크리스트에만 사용

### 6. 인과 관계 완전성
- "A이므로 C이다" 형태에서 중간 단계 B가 생략되지 않았는지 확인
- 글쓴이에게 자명한 전제가 독자에게도 자명한지 점검
- 새로운 개념(ARP, docker0, veth 등)이 처음 등장할 때 해당 지점에서 설명이 제공되는지 확인

## 다른 스킬과의 경계

이 스킬은 **구조적 뼈대**만 다룹니다:

- `/clarify-writing`은 **설명의 완성도**(빠진 설명 추가, 암묵적 전제 해소)를 다룹니다 → 이 스킬은 **문장/문단의 구조적 명확성**을 봅니다
- `/refine-style`은 **톤과 서사 구조**(종결어미, 서사적 도입)를 다룹니다 → 이 스킬은 **용어 일관성과 문장 간결함**을 봅니다
- `/smooth-flow`는 **읽기 자연스러움**(수식어 체인, 문맥 연결)을 다룹니다 → 이 스킬은 **주어-서술어 호응과 문단 구조**를 봅니다

## 글 구조 패턴

### 도입부
- 문제 제시나 배경 서사로 시작
  - 예: "정보를 멀리 보내려면 새로운 기술이 필요했습니다", "디지털이 선택된 이유가 있습니다", "패킷이 목적지를 찾아가는 과정에는 여러 단계가 필요합니다"
- 시리즈 글은 이전 글 참조로 연결
  - 예: "[Part 1](/dev/network/...)에서 ~를 살펴보았습니다"

### 섹션 구조
- `---`로 주요 섹션 구분
- `<br>`로 문단 간 여백 (중요한 전환점에서)
- `###`으로 하위 섹션 구분

### 마무리
```markdown
## 마무리

이 글에서 살펴본 내용을 정리하면:

- 핵심 포인트 1
- 핵심 포인트 2
- 핵심 포인트 3

<br>

[Part 2](/dev/network/...)에서는 ~를 다룹니다.

<br>

---

**관련 글**
- [글 제목](/경로/)
```

## 용어 표기 규칙

### 새 용어 도입
- 첫 등장: **굵은 글씨(영어 원어)**
  - 예: **전자기파(Electromagnetic Wave)**, **변조(Modulation)**
- 이후: 한글만 사용

### 약어 처리
- 첫 등장: 풀네임(약어)
  - 예: "TCP(Transmission Control Protocol)"
- 이후: 약어만 사용
- 익숙하지 않은 약어는 풀어서 설명
  - 예: "ISP" → "KT나 SK 같은 인터넷 회사"

## 작업 프로세스

### 1단계: 파일 읽기
- `$ARGUMENTS`가 있으면 해당 파일 읽기
- 없으면 사용자에게 파일 경로 또는 텍스트 요청

### 2단계: 구조 분석
- 도입부 형식 확인 (문제 제시/배경 서사로 시작하는가?)
- 섹션 구분 확인 (`---`, `###` 사용)
- 마무리 형식 확인 (요약, 다음 글 연결)

### 3단계: 원칙별 점검
- 각 수정 원칙에 맞지 않는 부분 식별
- 용어 일관성 확인 (같은 개념에 다른 용어 사용 여부)

### 4단계: 수정
- 가이드라인 적용하여 수정

### 5단계: 검토
- 원래 의미가 훼손되지 않았는지 확인
- 새로운 오류가 생기지 않았는지 확인

### 6단계: 출력
- 수정된 글 전체
- 주요 변경 사항 요약

긴 글은 섹션별로 나누어 서브에이전트에게 병렬로 수정을 요청할 수 있습니다.

### 서브에이전트 프롬프트
```
다음 글의 골격을 가이드라인에 따라 수정해주세요.

## 수정 원칙
1. 용어 일관성: 같은 개념에 같은 용어, 새 용어는 첫 등장 시 정의
2. 간결함: 불필요한 반복 제거, 필요한 정보 유지
3. 문장 명확성: 짧은 문장, 주어-서술어 명확
4. 문단 구조: 한 문단 한 주제, 앞뒤 연결

## 용어 표기
- 새 용어: **굵은 글씨(영어 원어)**
- 약어: 첫 등장 시 풀네임(약어)

## 수정할 글
[섹션 내용]

## 출력
수정된 글 + 변경 사항 요약
```
