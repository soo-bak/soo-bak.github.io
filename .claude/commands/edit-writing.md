# 글쓰기 수정 스킬

주어진 글을 아래 가이드라인에 따라 수정합니다.

## 인자
- `$ARGUMENTS`: 수정할 파일 경로 (선택사항). 없으면 사용자에게 파일 또는 텍스트를 요청합니다.

---

## 수정 원칙

### 1. 문장의 자연스러움
- 교과서적 표현 피하기: "~라는 것입니다" → 직접 서술
- "중요한 점은 ~입니다" → 바로 내용 서술
- 소리 내어 읽었을 때 자연스러워야 함

### 2. 직관적인 이해
- 한 문장에 하나의 개념
- 정의 → 예시 → 의미/결과 순서
- 이유 없이 결론만 제시하지 않기
- 독자가 추론해야 하는 논리적 간극 제거

### 3. 간결하지만 완전한 설명
- 불필요한 반복 제거
- 필요한 정보는 반드시 포함
- "없어도 의미 전달에 지장 없는 단어" 제거

### 4. 문장 구조
- 짧은 문장 선호 (2줄 이내)
- 주어-서술어 관계 명확
- 대명사보다 명확한 지시어

### 5. 객관적 문체
- 주관적 표현 삼가: "놀라운", "중요한", "매우"
- 비유 삼가: "마치 ~처럼"
- 강조는 구조로 (별도 문단, 굵은 글씨)

### 6. 문단 구조
- 하나의 문단에 하나의 주제
- "첫째", "둘째"로 논리 구분
- 앞뒤 문단 연결 명시

### 7. 단어 선택
- 같은 개념에 같은 용어 일관 사용
- 새 용어는 첫 등장 시 정의
- 자연스러운 조합: 파장-길다/짧다, 주파수-높다/낮다

### 8. 글의 흐름과 맥락
- 앞에서 설명한 개념이 뒤에서 활용
- 트레이드오프 관계 명시
- "앞서 살펴본 것처럼" 등으로 연결

### 9. 정확성
- 인과 관계가 논리적으로 맞는지 확인
- 불확실하면 조사 후 작성
- 상관관계를 인과관계로 서술하지 않기

## 글 구조 패턴

### 도입부
- 질문이나 문제 제시로 시작
  - 예: "정보를 멀리 보내려면", "왜 디지털인가", "패킷은 어떻게 목적지를 찾아가는가"
- 시리즈 글은 이전 글 참조로 연결
  - 예: "[Part 1](/dev/network/...)에서 ~를 살펴보았습니다"

### 섹션 구조
- `---`로 주요 섹션 구분
- `<br>`로 문단 간 여백 (중요한 전환점에서)
- `###`으로 하위 섹션 구분

### 마무리
```markdown
## 마무리

이 글에서 살펴본 내용을 정리하면:

- 핵심 포인트 1
- 핵심 포인트 2
- 핵심 포인트 3

<br>

[Part 2](/dev/network/...)에서는 ~를 다룹니다.

<br>

---

**관련 글**
- [글 제목](/경로/)
```

## 용어 표기 규칙

### 새 용어 도입
- 첫 등장: **굵은 글씨(영어 원어)**
  - 예: **전자기파(Electromagnetic Wave)**, **변조(Modulation)**
- 이후: 한글만 사용

### 약어 처리
- 첫 등장: 풀네임(약어)
  - 예: "TCP(Transmission Control Protocol)"
- 이후: 약어만 사용
- 익숙하지 않은 약어는 풀어서 설명
  - 예: "ISP" → "KT나 SK 같은 인터넷 회사"

## 전환 패턴

### 질문으로 전환
```markdown
그런데 음성 신호를 직접 전송하려면 문제가 생깁니다.
```

```markdown
하지만 컴퓨터에서는 웹 브라우저, 이메일, 카카오톡 등 여러 프로그램이 동시에 실행됩니다.
도착한 패킷이 어느 프로그램을 위한 것인지 구분해야 합니다.
```

### 앞 내용 연결
```markdown
앞서 살펴본 것처럼, 높은 주파수는 파장이 짧아 회절이 잘 일어나지 않습니다.
```

```markdown
[Part 1](/dev/network/NetworkCommunication-1/)에서 물리적 신호 전송의 원리를 살펴보았습니다.
```

### 연결 기법
- **명시적 연결**: "앞서 살펴본 회절 때문에..."
- **구조적 연결**: 같은 예시(Wi-Fi, 5G)를 반복 사용
- **대조적 연결**: "하지만 이것은 장점만 있는 것이 아닙니다"

## 다이어그램 활용

### 언제 사용하는가
- 구조를 설명할 때 (프로토콜 스택, 패킷 구조)
- 과정이나 흐름을 보여줄 때
- 비교나 대조를 할 때

### ASCII 다이어그램 예시

**패킷/프레임 구조:**
```
┌─────────┬────────┬────────┬──────┬─────────────────┬─────┐
│프리앰블  │ 목적지  │ 출발지  │ 타입 │      데이터     │ FCS │
└─────────┴────────┴────────┴──────┴─────────────────┴─────┘
```

**네트워크 토폴로지:**
```
라우터 A ─── 라우터 B ─── 라우터 C
   │
10.0.0.0/8
```

**프로토콜 스택/캡슐화:**
```
응용:     [         HTTP 요청         ]
전송:     [TCP 헤더][   HTTP 요청     ]
네트워크: [IP 헤더][TCP 헤더][HTTP 요청]
```

**비교 다이어그램:**
```
정상 상태:
A ─── B ─── C ─── D
      │
      E

장애 발생:
A ─── B ─X─ C ─── D
      │
      E
```

## 피해야 할 표현과 대안

### 기본 표현

| Before | After |
|--------|-------|
| "~라는 것입니다" | 직접 서술 |
| "~것이었습니다" | "~했습니다" |
| "중요한 점은 ~입니다" | 바로 내용 서술 |
| "~라고 할 수 있습니다" | "~입니다" |
| "그런데 여기서 근본적인 질문이 생깁니다" | "하지만 자연스러운 질문이 따라옵니다" |
| "이것이 바로 ~입니다" | 제거하고 직접 설명 |

### 주관적/감정적 표현

| Before | After |
|--------|-------|
| "놀랍게도", "흥미롭게도" | 삭제 |
| "마치 ~처럼" | 실제 원리로 설명 |
| "매우", "상당히", "굉장히" | 삭제 또는 구체적 수치 |
| "놀라운", "중요한", "핵심적인" | 삭제하거나 사실로 보여주기 |

### 실제 수정 사례 (Routing-1)

| Before | After | 이유 |
|--------|-------|------|
| "라우팅의 역사에서 그 답을 찾을 수 있습니다" | "이 문제는 인터넷이 처음 만들어질 때부터 있었습니다" | 교과서적 표현 |
| "링크" | "회선" | 독자에게 익숙한 용어 |
| "ISP" | "KT나 SK 같은 인터넷 회사" | 약어 설명 |
| "한 번 설정하면 끝" | "라우터끼리 메시지를 주고받지 않으므로" | 극단적 표현 완화 |
| "~라는 사실이 수학적으로 드러난 순간이었습니다" | "~임을 수식이 보여주고 있었습니다" | 간결하게 |

### 추가해야 할 설명 유형
- **"왜?"에 대한 답**: 분산 라우팅을 선택한 이유 (중앙 제어 = 단일 장애점)
- **구체적 예시**: 가정용 공유기 → google.com 시나리오
- **상태 비교**: 정상 상태 vs 문제 상태 다이어그램

## 작업 프로세스

### 1단계: 파일 읽기
- `$ARGUMENTS`가 있으면 해당 파일 읽기
- 없으면 사용자에게 파일 경로 또는 텍스트 요청

### 2단계: 구조 분석
- 도입부 형식 확인 (질문/문제 제시로 시작하는가?)
- 섹션 구분 확인 (`---`, `###` 사용)
- 마무리 형식 확인 (요약, 다음 글 연결)

### 3단계: 원칙별 점검
- 각 수정 원칙에 맞지 않는 부분 식별
- 용어 일관성 확인 (같은 개념에 다른 용어 사용 여부)
- 다이어그램 필요 여부 판단

### 4단계: 수정
- 가이드라인 적용하여 수정
- 필요시 다이어그램 추가
- 전환 패턴으로 문단 연결

### 5단계: 검토
- 원래 의미가 훼손되지 않았는지 확인
- 새로운 오류가 생기지 않았는지 확인
- 전체 흐름이 자연스러운지 통독

### 6단계: 출력
- 수정된 글 전체
- 주요 변경 사항 요약

### 서브에이전트 프롬프트
```
다음 글을 가이드라인에 따라 수정해주세요.

## 수정 원칙
1. 자연스러운 문장: 교과서적 표현 피하고 직접 서술
2. 직관적 이해: 한 문장에 하나의 개념, 이유와 결과 명확히
3. 간결+완전: 불필요한 반복 제거, 필요한 정보 유지
4. 명확한 구조: 짧은 문장, 주어-서술어 명확
5. 객관적 문체: 주관적 표현/비유 삼가
6. 용어 일관성: 같은 개념에 같은 용어
7. 맥락 연결: 앞뒤 문단과 논리적 연결
8. 정확성: 인과 관계 확인, 사실 기반

## 용어 표기
- 새 용어: **굵은 글씨(영어 원어)**
- 약어: 첫 등장 시 풀네임(약어)

## 수정할 글
[섹션 내용]

## 출력
수정된 글 + 변경 사항 요약
```

## 체크리스트

### 문장 수준
- [ ] 소리 내어 읽었을 때 자연스러운가?
- [ ] 한 문장에 하나의 개념만 있는가?
- [ ] 주어-서술어 관계가 명확한가?
- [ ] 불필요한 수식어("매우", "중요한")가 없는가?

### 문단 수준
- [ ] 하나의 주제를 다루는가?
- [ ] 앞뒤 문단과 연결이 명시되어 있는가?
- [ ] "왜?"에 대한 답이 있는가?

### 글 전체 수준
- [ ] 도입부가 질문/문제로 시작하는가?
- [ ] 용어가 일관되게 사용되는가?
- [ ] 새 용어가 첫 등장 시 정의되는가?
- [ ] 마무리에 요약이 있는가?
- [ ] 다이어그램이 필요한 곳에 있는가?
