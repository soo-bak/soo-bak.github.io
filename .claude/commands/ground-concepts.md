# 개념 접지 스킬 — 접지 (Grounding)

주어진 글의 **추상적 개념을 독자가 아는 것에 연결**합니다. 기술 용어와 추상적 설명을 독자가 이미 경험한 것에서 시작하여 이해할 수 있도록 합니다.

이 스킬은 VPNTunnel-1 글 수정 과정에서 발견된 패턴을 정리한 것입니다.

## 인자
- `$ARGUMENTS`: 수정할 파일 경로 (선택사항). 없으면 사용자에게 파일 경로를 요청합니다.

---

## 핵심 원칙

**"독자가 모른다고 가정하되, 바보라고 가정하지 않는다."**

독자는 기술 용어를 모를 수 있지만, 일상적인 경험은 가지고 있습니다. 그 경험에서 출발하여 새 개념으로 연결합니다.

---

## 점검 항목

---

## 1. 익숙한 것에서 시작하기

기술 용어를 바로 사용하지 않고, 독자가 이미 아는 것에서 시작합니다.

### 비교

```
# Before
서버나 라우터에는 네트워크 인터페이스가 있습니다.
흔히 보는 eth0, eth1 같은 것들입니다.
→ 독자: "eth0이 뭔데? 네트워크 인터페이스가 뭔데?"

# After
컴퓨터 뒷면에는 랜선을 꽂는 포트가 있습니다.
운영체제는 이 포트를 **네트워크 인터페이스**라고 부르고,
Linux에서는 eth0, eth1처럼 이름을 붙입니다.
→ 눈에 보이는 것(랜선 포트) → 용어(네트워크 인터페이스) → 구체적 이름(eth0)
```

### 수정 전략

1. 기술 용어가 처음 등장하는 곳을 찾습니다
2. 독자가 이미 본 적 있거나 경험한 것을 떠올립니다
3. 그 경험에서 시작하여 기술 용어로 연결합니다

### 패턴

| 기술 용어 | 익숙한 시작점 |
|----------|-------------|
| 네트워크 인터페이스 | 랜선 꽂는 포트 |
| MTU | 한 번에 보낼 수 있는 최대 크기 |
| 캡슐화 | 편지를 다른 봉투에 넣어 보내는 것 |
| 라우팅 테이블 | "어디로 가려면 어느 길로 가라"는 규칙 목록 |
| VPN 게이트웨이 | 네트워크 출입구에 있는 장비 |

### 점검 항목
- [ ] 기술 용어가 설명 없이 바로 등장하는 곳이 있는가?
- [ ] "흔히 보는", "잘 알려진" 같은 표현이 독자에게 실제로 익숙한 것을 가리키는가?
- [ ] 추상적 위치(예: "네트워크 경계")가 구체적으로 설명되는가?

## 2. 일관된 시나리오 유지

글 전체에서 하나의 구체적 시나리오를 유지하고, 모든 개념을 이 시나리오에 연결합니다.

### 비교

```
# Before
Site-to-Site VPN은 두 네트워크를 연결합니다.
→ 추상적

# After
서울 본사(10.1.0.0/16)와 부산 지사(10.2.0.0/16)를
연결하는 시나리오를 다시 떠올려봅시다.
→ 앞서 설정한 구체적 시나리오에 연결
```

### 수정 전략

1. 글 초반에 구체적 시나리오를 설정합니다 (예: 서울 본사 ↔ 부산 지사)
2. 새 개념을 설명할 때마다 이 시나리오에 적용합니다
3. "앞서 살펴본", "다시 떠올려봅시다" 등으로 시나리오를 상기시킵니다

### 시나리오 요소

| 요소 | 구체적 값 |
|------|----------|
| 본사 내부 네트워크 | 10.1.0.0/16 |
| 지사 내부 네트워크 | 10.2.0.0/16 |
| 본사 공인 IP | 203.0.113.5 |
| 지사 공인 IP | 198.51.100.5 |
| 본사 직원 PC | 10.1.0.5 |
| 지사 서버 | 10.2.0.5 |

### 점검 항목
- [ ] 글 초반에 구체적 시나리오가 설정되어 있는가?
- [ ] 새 개념이 등장할 때 시나리오에 적용되는가?
- [ ] 다이어그램에 시나리오의 구체적 값(IP 주소 등)이 포함되어 있는가?

## 3. 다이어그램 명확성

다이어그램은 구조가 아니라 **시나리오**를 보여줍니다.

### 비교: 구조 vs 시나리오

```
# Before (구조만 보여줌)
TCP/UDP 패킷:
┌─────────────────────────────────────┐
│ IP 헤더 (프로토콜: 6 또는 17)       │
│ TCP/UDP 헤더 (출발 포트, 도착 포트) │
│ 데이터                              │
└─────────────────────────────────────┘
→ 구조는 보여주지만, "그래서 뭐?"라는 질문에 답하지 않음

# After (시나리오로 문제를 보여줌)
TCP 연결 (NAT가 구분 가능):

내부 호스트 A (10.0.0.1:50001) ─┐
                               ├─► NAT ─► 인터넷 ─► 서버
내부 호스트 B (10.0.0.2:50002) ─┘

돌아오는 패킷:
  서버 → NAT (도착 포트 50001) → 호스트 A
  서버 → NAT (도착 포트 50002) → 호스트 B
→ 실제 상황에서 어떻게 동작하는지 보여줌
```

### 비교: 복잡한 흐름 vs 수평 흐름

```
# Before (방향이 혼란스러움)
본사 직원 PC                    본사 게이트웨이                      인터넷
10.1.0.5                                                              │
    │                                                                 │
    │  원본: 10.1.0.5 → 10.2.0.5                                     │
    │─────────────────────►  라우팅: tunnel0으로                      │
                                    │                                 │
                                    ▼                                 │

# After (왼쪽에서 오른쪽으로 일관된 흐름)
본사 직원 PC        본사 게이트웨이          인터넷           부산 게이트웨이
(10.1.0.5)         (203.0.113.5)                            (198.51.100.5)
     │                   │                                        │
     │  10.1.0.5 →       │                                        │
     │  10.2.0.5         │                                        │
     │──────────────────►│                                        │
                         │  캡슐화                                │
                         │  203.0.113.5 → 198.51.100.5           │
                         │───────────────────────────────────────►│
```

### 수정 전략

1. 다이어그램이 "구조"만 보여주는지, "시나리오"를 보여주는지 확인
2. 복잡한 방향 전환을 왼쪽→오른쪽 수평 흐름으로 단순화
3. 구체적 IP 주소, 포트 번호 등을 포함
4. 다이어그램 아래에 각 단계를 설명하는 텍스트 추가

### 점검 항목
- [ ] 다이어그램이 실제 시나리오를 보여주는가?
- [ ] 흐름이 왼쪽에서 오른쪽으로 일관적인가?
- [ ] 구체적 값(IP, 포트 등)이 포함되어 있는가?
- [ ] 다이어그램만 봐도 무슨 일이 일어나는지 이해할 수 있는가?

## 4. 불릿 리스트를 서사로 전환

"특징:", "해결책:", "사용 사례:" 같은 불릿 리스트를 서사적 설명으로 전환합니다.

### 비교

```
# Before
특징:
- 양쪽 네트워크가 하나의 사설 네트워크처럼 동작
- 네트워크 단위로 연결하므로 개별 장치마다 VPN 설정이 필요 없음

주요 사용 사례:
- 본사와 지사 간 상시 연결
- 여러 데이터센터 간 백본 연결
→ 정보는 있지만 "왜 그런지"가 없음

# After
본사 직원이 지사 서버(10.2.0.5)에 접근하면, 패킷은 본사 게이트웨이에서
암호화되어 터널을 통과하고, 지사 게이트웨이에서 복호화됩니다.
직원의 PC에는 VPN 소프트웨어가 필요 없습니다.
게이트웨이가 모든 처리를 대신하기 때문입니다.
→ 특징을 시나리오 속에서 자연스럽게 설명
```

### 수정 전략

1. "특징:", "해결책:" 등의 불릿 리스트를 찾습니다
2. 각 항목이 "왜 그런지"를 생각합니다
3. 시나리오 속에서 자연스럽게 설명하는 문장으로 전환합니다

### 점검 항목
- [ ] "특징:", "해결책:", "사용 사례:" 같은 불릿 리스트가 있는가?
- [ ] 불릿 항목이 "왜 그런지" 설명 없이 나열만 되어 있는가?
- [ ] 시나리오 속에서 자연스럽게 녹여낼 수 있는가?

## 5. 해결책과 문제의 명시적 연결

"이것이 해결책이다"라고 말할 때, 앞서 제시한 문제와 명시적으로 연결합니다.

### 비교

```
# Before
가장 간단한 방법은 터널 인터페이스의 MTU를 미리 낮추는 것입니다.
터널 인터페이스의 MTU를 1476으로 설정하면...
→ 왜 이것이 "해결책"인지 연결이 없음

# After
앞서 본 문제를 다시 봅시다. 1500바이트 패킷이 터널에 들어가면
캡슐화 후 1524바이트가 되어 MTU를 초과했습니다.

그렇다면 **처음부터 1476바이트 이하의 패킷만 터널에 들어가게 하면** 어떨까요?
→ 문제를 상기시키고, 해결 아이디어를 질문 형태로 제시
```

### 수정 전략

1. "해결책", "방법", "이렇게 하면 된다" 등의 표현을 찾습니다
2. 바로 앞에서 어떤 문제를 다뤘는지 확인합니다
3. "앞서 본 문제를 다시 봅시다" 등으로 문제를 상기시킵니다
4. 해결책이 문제를 어떻게 해결하는지 명시적으로 연결합니다

### 점검 항목
- [ ] 해결책이 제시될 때, 어떤 문제를 해결하는지 명시되어 있는가?
- [ ] "왜 이것이 해결책인가?"에 대한 답이 있는가?
- [ ] 문제 → 해결책의 논리적 연결이 명확한가?

## 6. 대조 예시 추가

개념을 설명할 때, "이것은 포함되고, 저것은 포함되지 않는다"를 보여주는 대조 예시를 추가합니다.

### 비교

```
# Before
부산 지사 대역(10.2.0.0/16)으로 가는 패킷은 tunnel0 인터페이스로 전송됩니다.
→ 터널로 가는 경우만 설명

# After
본사 직원이 부산 지사 서버(10.2.0.5)에 접속하면... tunnel0으로 전송됩니다.

그렇다면 본사 직원이 Google(8.8.8.8)에 접속하면 어떻게 될까요?
8.8.8.8은 부산 지사 대역(10.2.0.0/16)이 아닙니다.
패킷은 터널을 거치지 않고 eth1을 통해 인터넷으로 직접 나갑니다.

즉, **부산 지사로 가는 트래픽만 터널을 통과**하고,
나머지 인터넷 트래픽은 평소처럼 직접 나갑니다.
→ 터널을 통과하는 것 vs 통과하지 않는 것을 대비
```

### 수정 전략

1. 개념이나 규칙을 설명하는 곳을 찾습니다
2. "그렇다면 ~하면 어떻게 될까요?"로 대조 상황을 제시합니다
3. 대조를 통해 개념의 경계를 명확히 합니다

### 점검 항목
- [ ] 규칙/개념이 적용되는 경우만 설명하고 있는가?
- [ ] 적용되지 않는 경우(대조 예시)가 있으면 이해가 더 명확해지는가?
- [ ] "모든 트래픽이 ~한다"처럼 오해할 수 있는 곳이 있는가?

## 7. 기술적 정확성 검증

기술적 설명이 정확한지 확인하고, 단순화가 오해를 유발하지 않는지 검증합니다.

### 비교

```
# Before
TCP는 이 MTU를 보고 데이터를 더 작은 단위로 나눠서 보냅니다.
→ 기술적으로 부정확. TCP는 MTU를 직접 보지 않음

# After
TCP 연결을 맺을 때, 양쪽은 **MSS(Maximum Segment Size)**를 교환합니다.
MSS는 "한 번에 보낼 수 있는 TCP 데이터의 최대 크기"입니다.
이 값은 인터페이스의 MTU에서 IP 헤더(20바이트)와 TCP 헤더(20바이트)를 뺀 값입니다.
→ 실제 동작을 정확하게 설명
```

### 점검 항목
- [ ] 단순화된 설명이 기술적으로 틀린 내용을 담고 있는가?
- [ ] "A가 B를 한다"고 했을 때, 실제로 A가 그 동작을 수행하는가?
- [ ] 인과 관계가 정확한가? (상관관계를 인과관계로 서술하지 않았는가?)

## 8. 관련 글 참조 추가

개념이 다른 글에서 더 자세히 설명되어 있으면 링크를 추가합니다.

### 비교

```
# Before
이것이 라우팅 테이블입니다.
→ 라우팅 테이블을 처음 보는 독자는 막막함

# After
이것이 [라우팅 테이블](/dev/network/Routing-1/#라우팅-테이블-경로의-지도)입니다.
→ 더 알고 싶은 독자는 링크를 따라갈 수 있음
```

### 수정 전략

1. 개념이 처음 등장하는 곳을 찾습니다
2. 해당 개념을 더 자세히 다루는 기존 글이 있는지 확인합니다
3. 있으면 해당 섹션으로의 링크를 추가합니다

### 점검 항목
- [ ] 이 글에서 간략히 다루는 개념 중, 다른 글에서 자세히 설명된 것이 있는가?
- [ ] 독자가 배경 지식을 얻을 수 있는 참조가 제공되는가?

## 작업 프로세스

### 1단계: 파일 읽기 및 시나리오 파악
- `$ARGUMENTS`가 있으면 해당 파일 읽기
- 없으면 사용자에게 파일 경로 요청
- 글에서 사용하는 주요 시나리오(예시) 파악

### 2단계: 기술 용어 스캔
- 설명 없이 등장하는 기술 용어 식별
- 각 용어에 대해 "독자가 이미 아는 것"을 찾기
- 익숙한 것 → 기술 용어 순서로 재구성

### 3단계: 시나리오 일관성 확인
- 글 초반에 시나리오가 설정되어 있는지 확인
- 새 개념이 시나리오에 연결되는지 확인
- 다이어그램에 시나리오의 구체적 값이 있는지 확인

### 4단계: 다이어그램 점검
- 구조만 보여주는 다이어그램을 시나리오로 전환
- 복잡한 방향을 수평 흐름으로 단순화
- 구체적 값 추가

### 5단계: 불릿 리스트 및 해결책 점검
- 불릿 리스트를 서사로 전환
- 해결책과 문제의 연결 확인
- 대조 예시 추가

### 6단계: 기술적 정확성 검증
- 단순화된 설명이 오해를 유발하지 않는지 확인
- 필요시 WebSearch로 검증

## 체크리스트

### 용어와 개념
- [ ] 기술 용어가 익숙한 것에서 시작하여 도입되는가?
- [ ] 추상적 표현(예: "네트워크 경계")이 구체적으로 설명되는가?
- [ ] "흔히 보는"이 실제로 독자에게 익숙한 것을 가리키는가?

### 시나리오
- [ ] 글 전체에서 일관된 시나리오가 유지되는가?
- [ ] 새 개념이 시나리오에 적용되어 설명되는가?
- [ ] 다이어그램에 구체적 값(IP, 포트 등)이 있는가?

### 다이어그램
- [ ] 구조가 아닌 시나리오를 보여주는가?
- [ ] 흐름이 왼쪽→오른쪽으로 일관적인가?
- [ ] 다이어그램만 봐도 이해할 수 있는가?

### 설명 방식
- [ ] 불릿 리스트가 서사로 전환되었는가?
- [ ] 해결책이 문제와 명시적으로 연결되는가?
- [ ] 대조 예시가 개념의 경계를 명확히 하는가?

### 정확성
- [ ] 기술적 설명이 정확한가?
- [ ] 관련 글 참조가 추가되었는가?
