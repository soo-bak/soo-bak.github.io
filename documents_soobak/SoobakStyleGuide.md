# soobak 스타일 글 작성 가이드

이 문서는 `algorithm/boj/_posts` 글을 기준으로 한 soobak 스타일을 정리한 가이드입니다.  
목표는 **직관적이되 간결하며 완전한 설명**을 유지하면서, 형식과 코드 스타일을 일관되게 맞추는 것입니다.

## 1) 기본 철학
- 짧고 명확한 설명: 핵심 아이디어를 2~5단락으로 나눠 압축합니다.
- 단계적 전개: "먼저/다음으로/이후/마지막으로"처럼 순차 흐름을 드러냅니다.
- 과장/수사 최소화: 사실 중심으로, 불필요한 수식어를 피합니다.
- 문제 설명은 1~3문장으로 끝냅니다.

## 2) 파일 구조 (필수 섹션 순서)
1. Front matter (YAML)
2. `## 문제 링크`
3. `## 설명`
4. `<br>`
5. `## 접근법`
6. `<br>` (필요 시)
7. `- - -` (구분선)
8. `## Code`
9. `### C#`
10. `### C++`

### 2.1 Front matter 템플릿
```yaml
---
layout: single
title: "[백준 {문제번호}] {문제명} (C#, C++) - soo:bak"
date: "YYYY-MM-DD HH:MM:SS +0900"
description: 한 줄 요약
---
```
- `description`은 한 줄 요약이며 마침표를 붙이지 않습니다.
- `title`은 문제명과 언어 표기를 포함합니다.

### 2.2 문제 링크
```md
## 문제 링크
[12345번 - 문제명](https://www.acmicpc.net/problem/12345)
```

### 2.3 설명
- 문제를 1~3문장으로 요약합니다.
- 문장 끝은 보통 "~문제입니다." 형태가 자연스럽습니다.
- 긴 배경 설명이나 예시는 피합니다.

```md
## 설명
...문제입니다.

<br>
```

### 2.4 접근법
- 핵심 아이디어, 상태/자료구조, 진행 순서를 간결히 설명합니다.
- 필요하면 시간 복잡도를 마지막에 1문장으로 추가합니다.
- 문단은 2~5개 정도가 적당합니다.

```md
## 접근법
먼저 ...

다음으로 ...

이후 ...

<br>

- - -
```

### 2.5 Code 섹션
- C# 먼저, C++ 다음 순서.
- 각 언어는 표준 마크다운 코드 블록을 사용합니다.

```md
## Code

### C#
```csharp
// ...
```

### C++
```cpp
// ...
```
```

## 3) 문체/톤 규칙
- 평서문 위주, 짧고 단정한 문장.
- 문제 해결 흐름이 자연스럽게 이어지도록 연결어를 사용합니다.
- 불필요한 강조(느낌표, 이모지, 과장 표현) 금지.
- 핵심만 설명하고 세부 구현은 코드에 맡깁니다.

## 4) 접근법 서술 체크리스트
- 문제의 핵심 조건/제약을 한 문장으로 짚었는가?
- 어떤 알고리즘/자료구조를 쓰는지 명시했는가?
- 왜 그 방식이 맞는지 간단히 설명했는가?
- 구현 흐름이 순서대로 읽히는가?
- 필요 시 복잡도 한 줄을 추가했는가?

## 5) 코드 스타일 가이드 (공통)
- 들여쓰기 2칸.
- K&R 브레이스 스타일(`if (...) {`).
- 불필요한 주석은 최소화.
- 변수명은 짧고 직관적(`n, m, a, b, ans, sum, cnt` 등).
- 입출력은 빠른 방식 우선(C++ sync off, C# StringBuilder 등).
- 한 줄로 끝나는 조건/반복은 `{}`를 생략합니다.
- 여러 줄 블록은 `{}`로 감싸고, 블록 내부는 줄 단위로 분리합니다.
- `else if`는 같은 줄에 이어 씁니다.

## 6) C# 스타일 (최근 soobak 글 기준)
- 기본 구조:
  - `class Program` + `static void Main()` 형태 고정.
  - `namespace`는 사용하지 않습니다.
- 입력:
  - 한 줄: `Console.ReadLine()!.Split()`.
  - 전체 입력: `Console.In.ReadToEnd().Split()`.
  - 반복 입력은 `while ((line = Console.ReadLine()) != null)` 또는 `while (true)` + 종료 조건 방식.
- 출력:
  - 단일: `Console.WriteLine`.
  - 다중: `StringBuilder` 후 `Console.Write`.
- `var`를 기본으로 사용하고, 타입이 애매하거나 널 처리 필요 시에만 명시 타입 사용(`string? line;`).
- `LINQ`는 지양하고 루프 기반 구현.

## 7) C++ 스타일 (최근 soobak 글 기준)
- 기본 구조:
  - `#include <bits/stdc++.h>`
  - `using namespace std;`
  - `ios::sync_with_stdio(false); cin.tie(nullptr);`
- 타입:
  - 기본은 `int` 위주.
  - 큰 범위가 필요한 경우에만 `long long` 또는 `typedef long long ll;`.
  - `auto`는 사용하지 않습니다.
- 입출력:
  - 보통 `cin`/`cout`, 줄 단위는 `getline` + 파싱.
  - 출력은 `cout << ... << "\n";` 형태.
- 별칭:
  - 자주 쓰는 타입에 한정해 `typedef`/`using` 사용.

## 8) 중괄호/개행 규칙
- 기본 원칙: 반복문/조건문은 `{}` 사용.
- 예외: 아주 짧은 단일 문장일 때만 1줄 if 허용.
  - 예: `if (cnt[i] > cnt[ans]) ans = i;`
- `else if`는 같은 줄에 이어 씁니다.

## 9) 자주 쓰는 표현 템플릿
- 문제 설명:
  - "…을/를 구하는 문제입니다."
  - "…일 때 …을 출력하는 문제입니다."
- 접근법:
  - "먼저 …을 구합니다."
  - "다음으로 …을 이용해 …합니다."
  - "이후 …을 반복하며 …을 갱신합니다."
  - "마지막으로 …을 출력합니다."

## 10) 최종 점검
- 섹션 순서와 제목이 정확한가?
- `title`/`description`/링크가 문제와 일치하는가?
- 설명과 접근법이 간결하지만 빠진 논리가 없는가?
- C#과 C++ 코드가 같은 알고리즘을 구현하는가?
- 들여쓰기, 괄호, 입출력 스타일이 통일됐는가?
